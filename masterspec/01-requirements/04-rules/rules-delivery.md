# Правила: формирование и гранулярность доставки

## Бизнес-правила
- Единица доставки — батч одного диалога за такт сбора: одно письмо на диалог за цикл, внутри —
  упорядоченный список сообщений (от кого / когда / текст). Письмо-на-сообщение запрещено (защита
  от почтового флуда активного чата).
- Множество источников Telegram доставляется в ОДИН целевой ящик; каждое письмо помечено тегом
  источника/диалога (например `dm:<отправитель>`), маршрутизация ответа — по журналу связки.
  Мультиящик поддержан архитектурно, дефолт — много источников → один ящик.
- Изображения из Telegram встраиваются в тело письма (инлайн), а не прикладываются файлом.
- Прочие вложения — по размеру: не превышающие порог из конфигурации приходят приложенным файлом;
  превышающие порог — текстовым указанием (имя / размер / тип) без тела. Порог настраивается.
  (Задел версии 2 — ссылка на веб-диск для скачивания крупных вложений.)

## Инженерные правила
- Тело письма — HTML (со встроенными изображениями) и обязательным текстовым запасным вариантом.
- Порядок сообщений внутри батча сохраняется (хронологический).
- Тег источника однозначно идентифицирует диалог для последующей маршрутизации ответа.

## Внешние ограничения (если есть)
- Нет.

## Принципы принятия решений
- Приоритет — обозримость почты: группировать агрессивно (диалог за такт), не дробить.
- Крупные бинарные данные не раздувают ящик: указание вместо тела при превышении порога.

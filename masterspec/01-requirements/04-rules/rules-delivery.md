# Правила: формирование и гранулярность доставки

## Бизнес-правила
- Единица доставки — батч одного диалога за такт сбора: одно письмо на диалог за цикл, внутри —
  упорядоченный список сообщений (от кого / когда / текст). Письмо-на-сообщение запрещено (защита
  от почтового флуда активного чата).
- Множество источников Telegram доставляется на ОДИН адрес пользователя (U); отправка — через ящик
  моста (B, SMTP). Каждое письмо помечено тегом источника/диалога (например `dm:<отправитель>`),
  маршрутизация ответа — по журналу связки. Мультиадрес поддержан архитектурно, дефолт — много
  источников → один адрес U.
- Изображения из Telegram встраиваются в тело письма (инлайн), а не прикладываются файлом.
- Каждое сообщение в письме представляется максимально близко к виду в клиенте Telegram: автор
  (отображаемое имя и @username), локализованные дата/время, сохранённое форматирование
  (полужирный / курсив / ссылки / моноширинный / списки), показанная цитата-ответ (на что и кому
  отвечено), медиа — инлайн (-> fn-media-in-email). Форматирование не деградирует в plain-кашу.
- Каждое сообщение в письме несёт глубокую ссылку (deep-link) на оригинал в Telegram — переход к
  тому самому сообщению или топику в один тап.
- Прочие вложения — по размеру: не превышающие порог из конфигурации приходят приложенным файлом;
  превышающие порог — текстовым указанием (имя / размер / тип) без тела. Порог настраивается.
  (Задел версии 2 — ссылка на веб-диск для скачивания крупных вложений.)

## Инженерные правила
- Тело письма — HTML (со встроенными изображениями) и обязательным текстовым запасным вариантом.
- Порядок сообщений внутри батча сохраняется (хронологический).
- Тег источника однозначно идентифицирует диалог для последующей маршрутизации ответа.
- Глубокая ссылка строится по типу источника: приватные — `t.me/c/<chat>/<msg>`; публичные
  (пользователь / канал) — `t.me/<user|channel>/<msg>`; при необходимости — схема `tg://`; для топика
  ссылка ведёт в соответствующий топик.
- Форматирование Telegram передаётся средствами HTML; при невозможности отрисовать элемент он
  деградирует в текст без потери содержания.

## Внешние ограничения (если есть)
- Нет.

## Принципы принятия решений
- Приоритет — обозримость почты: группировать агрессивно (диалог за такт), не дробить.
- Крупные бинарные данные не раздувают ящик: указание вместо тела при превышении порога.

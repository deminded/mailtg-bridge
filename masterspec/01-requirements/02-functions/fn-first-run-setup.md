# Функция: Первичная авторизация Telegram и порождение сессии

## Цель
Однократно, интерактивно авторизовать доступ к аккаунту Telegram и создать файл сессии, чтобы
фоновый сервис далее работал неинтерактивно (гибридная авторизация).

## Инициатор
Пользователь (запуск интерактивной команды инициализации).
- **Внешний инициатор/канал:** Пользователь — интерактивный запуск CLI (телефон/код/2FA) против
  Telegram API (внешняя система авторизации, MTProto); при отзыве сессии в фоне — письмо-уведомление
  на адрес пользователя U (через ящик моста B, SMTP).

## Триггер
Первый запуск или отсутствие действительной сессии.

## Предусловия
У пользователя есть доступ к аккаунту Telegram (телефон для приёма кода); задана конфигурация
доступа и путь к сессии.

## Основной поток
1. Пользователь запускает интерактивную команду инициализации.
2. Система запрашивает телефон, код подтверждения и, при включённой 2FA, пароль.
3. Система создаёт файл сессии с правами доступа только для владельца, вне репозитория.
4. Фоновый сервис в дальнейшем использует сессию неинтерактивно.

## Альтернативные потоки
- Действительная сессия уже существует → интерактивный шаг не требуется, сервис стартует headless.
- Двухфакторная защита не включена → шаг ввода пароля пропускается.

## Исключительные потоки (ошибки, отказы)
- Неверный код или пароль → авторизация не завершается, файл сессии не создаётся; наблюдаемый исход:
  явная ошибка и предложение повторить.
- Сессия отозвана или недействительна во время работы → сервис переводится в состояние «сессия
  недействительна»: опрос Telegram останавливается, пользователю (U) уходит письмо-уведомление о
  необходимости повторной интерактивной инициализации; повторного зацикливания на той же ошибке нет
  (блокирующее состояние до реинициализации) (-> cdm-bridge).

## Результат
Файл сессии, пригодный для неинтерактивной работы фонового сервиса.

## Постусловия
Сессия хранится с правами только для владельца, вне репозитория; сервис может стартовать headless.

## Контракт живой эксплуатации (OE)

### OE-LOAD — объём и ритм
- **Статус:** APPLICABLE
- **Контракт:** Разовая операция на развёртывание (население = 1); повторный запуск — только
  событийно, при недействительной/отозванной сессии (переавторизация) или явном `--reauthorize`;
  нет периодического или пикового потока — это не тактовая функция.
- **Основание:** измерение — интерактивная инициализация оформлена отдельной операционной подкомандой
  вне основного рабочего цикла, проверено на работающей системе.
- **Проверка:** AC-02, AC-03

### OE-INPUT — реальные формы входа
- **Статус:** APPLICABLE
- **Контракт:** Интерактивный ввод оператора через терминал (не сетевой формат): телефон в
  международном формате, код подтверждения из SMS/приложения Telegram, пароль 2FA при включённой
  защите (без эха в терминале). Telegram API — единственный внешний производитель кода подтверждения
  и решения о необходимости 2FA.
- **Основание:** измерение — интерактивный ввод телефона/кода/пароля 2FA (без эха пароля) и
  ветвление по требованию 2FA проверены на работающей системе.
- **Проверка:** AC-01

### OE-EVIDENCE — наблюдаемый исход и след
- **Статус:** APPLICABLE
- **Контракт:** Успех — создан файл сессии, физически проверяемый (права 600); отказ (неверный код/
  пароль) — явная ошибка в терминале, файл сессии не создаётся. Отзыв сессии во время работы —
  письмо-уведомление пользователю U (наблюдаемый пользователем факт вне терминала) плюс блокирующее
  состояние без повторного зацикливания на той же ошибке.
- **Основание:** измерение — переход в состояние «сессия недействительна» и отправка
  письма-уведомления при отзыве сессии проверены на работающей системе.
- **Проверка:** AC-01, AC-03

### OE-SOURCES — классы источников и политика
- **Статус:** N/A — источник команды инициализации не является входящим потоком сообщений с
  политикой допуска; единственный и неизбираемый актор — оператор развёртывания, вводящий код
  напрямую в интерактивной сессии Telegram API. Классов источников/белого списка здесь нет.
- **Контракт:** N/A
- **Основание:** измерение — единственный неизбираемый актор (оператор развёртывания) без списка
  допустимых отправителей подтверждён на работающей системе.
- **Проверка:** N/A

### OE-SECURITY — конфиденциальность и целостность
- **Статус:** APPLICABLE
- **Контракт:** Файл сессии — права доступа только для владельца (600), вне репозитория; секреты
  (`TG_API_HASH`, пароль 2FA при вводе) не логируются и не сохраняются в открытом виде за пределами
  сессии; поставка явно предупреждает о риске userbot (утрата сессии = полный доступ ко всему
  аккаунту). Общая политика — -> rules-security, -> nfr-security.
- **Основание:** измерение — права доступа только для владельца (600) на файл сессии и ввод пароля
  2FA без эха проверены на работающей системе.
- **Проверка:** AC-01

### OE-RESILIENCE — повтор, отказ и восстановление
- **Статус:** APPLICABLE
- **Контракт:** Недействительная/отозванная сессия переводит фоновый сервис в блокирующее состояние
  без повторного зацикливания на той же ошибке (уведомление отправляется один раз, флаг `notified`
  предотвращает спам); переавторизация (`--reauthorize`) создаёт новую сессию поверх; единственный
  экземпляр процесса действует и для setup-подкоманды (общий `LifetimeLock`). Общая формула —
  -> rules-integrity.
- **Основание:** измерение — однократное уведомление об отзыве сессии (без повторного зацикливания)
  и переавторизация поверх существующей сессии проверены на работающей системе.
- **Проверка:** AC-03

### OE-DELIVERY — сквозной путь результата
- **Статус:** APPLICABLE
- **Контракт:** Внешний путь результата этой функции для сценария отзыва сессии — письмо-уведомление
  «сессия недействительна», доставленное на адрес U; доставлено = SMTP-приём (тот же предел
  подтверждения, что и у остальных писем моста); повторных уведомлений на одну и ту же
  недействительность нет (не спам).
- **Основание:** измерение — письмо-уведомление о недействительности сессии и защита от повторных
  уведомлений на ту же недействительность проверены на работающей системе.
- **Проверка:** AC-03

### OE-CONTROL — живой управляющий ввод
- **Статус:** N/A — интерактивный ввод (телефон/код/пароль) — это протокол авторизации Telegram
  (MTProto), а не язык управления мостом; грамматика управления мостом (MAILTG ON/OFF) обрабатывается
  отдельной функцией.
- **Контракт:** N/A — грамматика управления принадлежит -> fn-bridge-control-by-email.
- **Основание:** измерение — интерактивная инициализация не содержит разбора текстовых команд, только
  протокол авторизации Telegram, подтверждено на работающей системе.
- **Проверка:** N/A

## Критерии приёмки
- AC-01: дано валидные телефон/код (и пароль при 2FA), когда выполнен интерактивный setup, тогда
  создаётся файл сессии с правами доступа только для владельца (600) вне репозитория.
- AC-02: дано действительная сессия, когда запускается фоновый сервис, тогда он стартует
  неинтерактивно, без запроса кода.
- AC-03: дано недействительная/отозванная сессия, когда сервис пытается работать, тогда опрос
  останавливается, пользователю (U) уходит письмо-уведомление и сервис НЕ зацикливается на той же
  ошибке (остаётся в блокирующем состоянии до повторной инициализации).

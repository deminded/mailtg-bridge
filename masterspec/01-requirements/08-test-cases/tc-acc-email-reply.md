# Приёмочный тест: ответ из почты в Telegram от лица пользователя

## Проверяет
- Функция: -> fn-email-reply-to-tg
- AC: AC-01, AC-02, AC-03, AC-04, AC-05, AC-06, AC-07, AC-08, AC-09, AC-10, AC-11
- Грани: -> fn-email-reply-to-tg/OE-LOAD, -> fn-email-reply-to-tg/OE-INPUT,
  -> fn-email-reply-to-tg/OE-EVIDENCE, -> fn-email-reply-to-tg/OE-SOURCES,
  -> fn-email-reply-to-tg/OE-SECURITY, -> fn-email-reply-to-tg/OE-RESILIENCE,
  -> fn-email-reply-to-tg/OE-DELIVERY

## Предусловия
Существует доставленное bridged-письмо диалога D с записью связки (отправлено с ящика моста B на
адрес пользователя U); мост включён; ящик B опрашивается каждые `SEND_INTERVAL_SECONDS` (дефолт 30 с).

## Шаги выполнения
1. **Действие:** Ответить с адреса U на bridged-письмо диалога D (top-post, без цитаты клиента);
   дождаться такта опроса ящика моста B.
   **Тестовые данные:** отправитель=U; получатель=B; in-reply-to=Message-ID bridged-письма D;
   текст="ok, понял"
   **Ожидаемый результат:**
   - Текст ответа появляется в диалоге D в Telegram от лица пользователя.
   - Ответ маршрутизирован по записи связки в диалог D (по диалогу, не по отдельному сообщению).

2. **Действие:** Ответить на произвольное не-bridged письмо (без записи связки); дождаться такта
   опроса.
   **Тестовые данные:** отправитель=U; получатель=B; in-reply-to=Message-ID письма без записи связки
   **Ожидаемый результат:**
   - В Telegram ничего не публикуется (ответ игнорируется).

3. **Действие:** Ответить на bridged-письмо диалога D с постороннего адреса X (≠ U); дождаться такта
   опроса; проверить журнал.
   **Тестовые данные:** отправитель=X (не U); in-reply-to=Message-ID bridged-письма D
   **Ожидаемый результат:**
   - В Telegram ничего не публикуется, факт логируется.

4. **Действие:** Ответить с адреса U на bridged-письмо, где ответ обёрнут HTML-цитатой клиента
   (Gmail `gmail_quote`, Outlook блок «Кому/От/Дата», русская атрибуция «… писал(а):»); дождаться
   такта опроса.
   **Тестовые данные:** тело письма = top-post + `<blockquote class="gmail_quote">`… с оригиналом;
   вариант 2: top-post + строка «Иван Петров писал(а):» + `>`-цитаты
   **Ожидаемый результат:**
   - В Telegram публикуется только top-post пользователя, без цитируемого хвоста.

5. **Действие:** Отправить письмо со признаком автоответа (заголовок `Auto-Submitted: auto-replied`
   ИЛИ `List-Id`, ИЛИ `Precedence: bulk`) в ответ на bridged-письмо D, формально «от» адреса U.
   **Тестовые данные:** From=U; Auto-Submitted=auto-replied; in-reply-to=Message-ID bridged-письма D
   **Ожидаемый результат:**
   - Письмо не считается доверенным ответом (анти-петля), в Telegram ничего не публикуется.

6. **Действие:** Ответить с адреса U на bridged-письмо D; проверить журнал такта на предмет текста
   ответа и секретов.
   **Тестовые данные:** лог такта после обработки ответа из шага 1
   **Ожидаемый результат:**
   - Журнал не содержит текста ответа и секретов.

7. **Действие:** Смоделировать Telegram FloodWait/временную ошибку на публикации ответа с адреса U
   на bridged-письмо D; дать такту завершиться; восстановить и дать следующий такт с тем же письмом
   всё ещё в ящике B.
   **Тестовые данные:** инъекция FloodWait на `post_as_user` для диалога D на один такт
   **Ожидаемый результат:**
   - Письмо НЕ помечается потреблённым (`consumed_mail`) на неуспешном такте.
   - На следующем такте ответ публикуется без потери (at-least-once), явного письма-уведомления об
     этом отказе не отправляется.

8. **Действие:** После успешной публикации ответа (шаг 1) выполнить повторный опрос ящика B / рестарт
   сервиса с тем же ответом-письмом всё ещё физически в ящике.
   **Тестовые данные:** повторный опрос/рестарт после шага 1
   **Ожидаемый результат:**
   - Ответ повторно в Telegram не публикуется (потреблён единожды).

9. **Действие:** Наполнить ящик B N письмами (bridged-ответы, не-bridged, спам), из которых только
   k требуют публикации; выполнить такт опроса; повторить такт без новых писем.
   **Тестовые данные:** N=50 писем в INBOX ящика B, k=3 необработанных
   **Ожидаемый результат:**
   - Каждый такт выполняет полную выборку `SEARCH ALL` по ящику (N писем).
   - Уже потреблённые письма пропускаются логически без повторной публикации.

## Постусловия
Ответ доставлен в исходный диалог; ответ без записи связки не порождает публикации.

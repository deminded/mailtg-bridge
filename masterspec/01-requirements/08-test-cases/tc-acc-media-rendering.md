# Приёмочный тест: представление медиа и вложений в письме

## Проверяет
- Функция: -> fn-media-in-email
- AC: AC-01, AC-02, AC-03, AC-04, AC-05, AC-06, AC-07
- Грани: -> fn-media-in-email/OE-LOAD, -> fn-media-in-email/OE-INPUT,
  -> fn-media-in-email/OE-EVIDENCE, -> fn-media-in-email/OE-SECURITY,
  -> fn-media-in-email/OE-RESILIENCE, -> fn-media-in-email/OE-DELIVERY

## Предусловия
Порог размера вложений задан в конфигурации значением P (`ATTACHMENT_THRESHOLD_BYTES`); лимит
письма — `EMAIL_SIZE_LIMIT_BYTES`. Оба ключа и их дефолты -> data-config.

## Шаги выполнения
1. **Действие:** Прислать сообщение с изображением; дождаться письма.
   **Тестовые данные:** вложение image/jpeg, 500 КБ
   **Ожидаемый результат:**
   - Изображение отображается встроенным в теле письма (не приложением).

2. **Действие:** Прислать сообщение с малым файлом (≤ P); дождаться письма.
   **Тестовые данные:** вложение application/pdf, размер = P − 1 байт
   **Ожидаемый результат:**
   - Малый файл приходит приложенным файлом.

3. **Действие:** Прислать сообщение с крупным файлом (> P); дождаться письма.
   **Тестовые данные:** вложение application/zip, размер = P + 1 байт
   **Ожидаемый результат:**
   - Крупный файл приходит текстовым указанием (имя/размер/тип) без тела.

4. **Действие:** Изменить порог P в конфигурации (`ATTACHMENT_THRESHOLD_BYTES`); повторить шаг 2 с
   файлом, ставшим крупным при новом пороге.
   **Тестовые данные:** ATTACHMENT_THRESHOLD_BYTES уменьшен ниже размера файла из шага 2
   **Ожидаемый результат:**
   - После смены порога граница файл/указание меняется без изменения кода.

5. **Действие:** Прислать сообщение с набором вложений, суммарный размер которых превысил бы лимит
   письма (`EMAIL_SIZE_LIMIT_BYTES`).
   **Тестовые данные:** батч из нескольких сообщений с вложениями, совокупный MIME-размер >
   EMAIL_SIZE_LIMIT_BYTES
   **Ожидаемый результат:**
   - Крупные вложения приходят текстовым указанием (или батч дробится на несколько писем).
   - Письмо доставляется, ничего не теряется молча.

6. **Действие:** Смоделировать сбой скачивания вложения из Telegram (MediaUnavailable/временная
   ошибка) для одного сообщения батча; дождаться письма.
   **Тестовые данные:** инъекция ошибки скачивания для одного вложения батча
   **Ожидаемый результат:**
   - Вложение заменяется текстовым указанием о недоступности («вложение недоступно»).
   - Письмо всё равно доставлено, текст сообщения не блокируется.

7. **Действие:** Проследить путь временного файла вложения от скачивания до отправки письма (успешной
   и неуспешной), проверить состояние временного каталога после такта.
   **Тестовые данные:** TEMP_DIR перед и после такта с вложением (ключ -> data-config)
   **Ожидаемый результат:**
   - Временный файл вложения удалён после отправки письма (успешной или нет), не остаётся в
     постоянном хранилище.

## Постусловия
Тело письма — HTML со встроенными изображениями и текстовым запасным вариантом.

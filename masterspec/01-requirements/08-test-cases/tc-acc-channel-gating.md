# Приёмочный тест: гейтинг каналов и групп (белый список/упоминание)

## Проверяет
- Функция: -> fn-channel-update-to-email
- AC: AC-01, AC-02, AC-03, AC-04, AC-05, AC-06, AC-07, AC-08, AC-09, AC-10, AC-11
- Грани: -> fn-channel-update-to-email/OE-LOAD, -> fn-channel-update-to-email/OE-INPUT,
  -> fn-channel-update-to-email/OE-EVIDENCE, -> fn-channel-update-to-email/OE-SOURCES,
  -> fn-channel-update-to-email/OE-SECURITY, -> fn-channel-update-to-email/OE-RESILIENCE,
  -> fn-channel-update-to-email/OE-DELIVERY

## Предусловия
В конфигурации: канал W — в белом списке; канал X — не в белом списке; политика упоминаний
включена; отслеживаемый набор источников = whitelist ∪ mention_list.

## Шаги выполнения
1. **Действие:** Опубликовать сообщение в W (вайтлист); дождаться такта сбора
   (`COLLECT_INTERVAL_SECONDS`).
   **Тестовые данные:** источник=W (в whitelist); COLLECT_INTERVAL_SECONDS=60
   **Ожидаемый результат:**
   - Сообщение из W доставлено письмом с тегом источника в пределах такта.

2. **Действие:** Опубликовать в X (невайтлист) сообщение с прямым упоминанием пользователя;
   дождаться такта.
   **Тестовые данные:** источник=X (не в whitelist); текст содержит `@username` пользователя
   **Ожидаемый результат:**
   - Сообщение с упоминанием из X доставлено письмом.

3. **Действие:** Опубликовать в X обычное сообщение без упоминания; дождаться такта.
   **Тестовые данные:** источник=X; текст без упоминания пользователя
   **Ожидаемый результат:**
   - Сообщение из X без упоминания не доставлено (письма нет) — поведение по умолчанию.

4. **Действие:** Убрать W из белого списка правкой конфигурации (без изменения кода); опубликовать
   в W; дождаться такта.
   **Тестовые данные:** WHITELIST_JSON изменён, W удалён из списка
   **Ожидаемый результат:**
   - После правки конфигурации W перестаёт доставляться — поведение изменилось без изменения кода.

5. **Действие:** Кликнуть глубокую ссылку доставленного сообщения из W (публичный) и из приватной
   группы/топика (участнику); сверить представление письма с видом в клиенте.
   **Тестовые данные:** сообщение из публичного W; сообщение из приватной группы с topic_id
   **Ожидаемый результат:**
   - Глубокая ссылка открывает именно это сообщение (публичные — `t.me/<user|channel>/<msg>`,
     приватные — `t.me/c/<chat>/<msg>`, топик — в свой топик).
   - Автор, локализованное время, форматирование и цитата-ответ видны как в клиенте без потери
     форматирования.

6. **Действие:** Настроить набор из нескольких отслеживаемых источников (whitelist ∪ mention_list) и
   смоделировать простой перед тактом; выполнить такт сбора.
   **Тестовые данные:** отслеживаемый набор из N источников, k из них с новыми сообщениями (k < N)
   **Ожидаемый результат:**
   - Фетчатся только источники с `top_id` выше сохранённого курсора (round-trips ≈ k, не N).
   - Backlog пересобирается без потери на последующих тактах.

7. **Действие:** Опубликовать сообщение от бота (`is_bot=true`) в вайтлист-канале W при
   `bot_policy=selected`, где @username бота отсутствует в `bot_list`; дождаться такта.
   **Тестовые данные:** источник=W (whitelist); отправитель — бот не из bot_list;
   BOT_POLICY=selected
   **Ожидаемый результат:**
   - Сообщение от бота не доставляется, несмотря на попадание W в белый список.

8. **Действие:** После такта с обработкой сообщений канала/группы проверить содержимое журнала (лог)
   сервиса.
   **Тестовые данные:** лог такта после шага 1
   **Ожидаемый результат:**
   - Журнал содержит диалог/тег источника/счётчики обработки, но не содержит текста сообщений и
     секретов (0 вхождений).

9. **Действие:** Смоделировать временную недоступность источника X (сетевая ошибка/FloodWait) на
   такте; дать такту завершиться; восстановить доступность и дать следующий такт.
   **Тестовые данные:** инъекция ошибки при опросе X на один такт
   **Ожидаемый результат:**
   - Такт для X пропускается без потери курсора, соблюдая backoff.
   - Сбор для X продолжается при восстановлении доступа.

10. **Действие:** Смоделировать неуспешную отправку письма-батча канала на такте; дать такту
    завершиться; восстановить отправку и дать следующий такт.
    **Тестовые данные:** инъекция ошибки в SmtpMailer.send для батча канала на один такт
    **Ожидаемый результат:**
    - Курсор диалога не продвинут на неуспешном такте.
    - На следующем такте сообщения доставлены без потери и без дубля.

## Постусловия
Доставлены только адресованные обновления; смена политики достигнута только конфигурацией.

## Предусловия
Чистая машина; доступ к аккаунту Telegram (телефон для кода); открытый репозиторий поставки.

## Шаги выполнения
1. **Действие:** Развернуть мост на чистой машине по README (клонировать репозиторий, установить
   зависимости, заполнить пример конфигурации), засекая время до первого доставленного письма.
   **Тестовые данные:** чистая VM; README поставки; .env.example
   **Ожидаемый результат:**
   - Развёртывание до первого письма укладывается в ≤ 10 минут.

2. **Действие:** Выполнить интерактивный setup (телефон/код, при 2FA — пароль), проверить создание
   файла сессии; затем перезапустить как фоновый сервис.
   **Тестовые данные:** валидные телефон/код (и пароль при 2FA)
   **Ожидаемый результат:**
   - После setup создан файл сессии; фоновый сервис стартует неинтерактивно, без запроса кода.

3. **Действие:** Инвалидировать сессию (отозвать авторизацию со стороны Telegram) и дать фоновому
   сервису выполнить очередной такт.
   **Тестовые данные:** отозванная/недействительная сессия
   **Ожидаемый результат:**
   - Опрос Telegram останавливается.
   - Пользователю (U) уходит письмо-уведомление о необходимости повторной инициализации.
   - Сервис НЕ зацикливается на той же ошибке (остаётся в блокирующем состоянии, повторного
     уведомления при следующих тактах нет).

4. **Действие:** Проверить права файла сессии и файлов секретов (учётные данные почты, TG_API_HASH).
   **Тестовые данные:** stat на путь TG_SESSION_PATH и файл окружения секретов
   **Ожидаемый результат:**
   - Права файла сессии и файлов секретов = 600.

5. **Действие:** Прогнать секрет-скан-инструмент по репозиторию поставки; проверить размещение
   сессии/секретов относительно дерева репозитория.
   **Тестовые данные:** секрет-скан репозитория (например, gitleaks/trufflehog)
   **Ожидаемый результат:**
   - Секрет-скан репозитория чист (0 находок).
   - Сессия и секреты размещены вне дерева репозитория.

6. **Действие:** Инспектировать логи сервиса после нескольких тактов доставки (личка + ответ +
   команда управления).
   **Тестовые данные:** лог после ≥3 тактов разных типов операций
   **Ожидаемый результат:**
   - Логи не содержат тел сообщений и секретов.

7. **Действие:** Сменить почтового провайдера правкой конфигурации (адрес/порт/креды IMAP/SMTP),
   не трогая код; перезапустить сервис.
   **Тестовые данные:** новые B_IMAP_HOST/B_SMTP_HOST/креды в конфигурации
   **Ожидаемый результат:**
   - Смена провайдера достигнута правкой конфигурации без изменения кода (0 изменённых файлов кода).

8. **Действие:** Открыть README и найти раздел рисков userbot; открыть раздел о заведении ящика
   моста B.
   **Тестовые данные:** README поставки
   **Ожидаемый результат:**
   - README содержит явный раздел о риске userbot и цене компрометации сессии.
   - README содержит раздел о заведении ящика моста B (выделенный аккаунт, app-пароль); пример
     конфигурации покрывает креды ящика B (IMAP/SMTP) и адрес пользователя U (100% обязательных
     параметров).

## Постусловия
Развёрнутый экземпляр работает; секреты и сессия защищены правами доступа и размещением вне
репозитория; провайдер сменяем конфигурацией.

# Приёмочный тест: анти-петля, дедуп и состояние через перезапуск

## Проверяет
- Функции: -> fn-dm-batch-to-email, -> fn-email-reply-to-tg
- AC: AC-03 (dm), AC-04 (dm), AC-03 (reply), AC-05 (reply)
<!-- Fold guardrail exceptions-first: терминальные состояния помечены, из терминала нет неявных выходов. -->

## Предусловия
Мост включён; есть история доставленных сообщений и запись связки диалога D.

## Входные данные
- Уже доставленные ранее сообщения диалога D.
- Смоделированный сбой отправки письма ИЛИ записи связки/продвижения курсора на одном такте.
- Ответ пользователя, опубликованный мостом в Telegram (собственное эхо).
- Один и тот же ответ-письмо пользователя, попавший в опрос ящика дважды (повторный опрос/перезапуск
  после обработки) — для идемпотентности приёма.

## Шаги
1. Перезапустить сервис и выполнить повторный опрос по всем диалогам.
2. На одном такте смоделировать неуспешную отправку письма (и отдельно — неуспешную запись
   связки/курсора при успешной отправке), затем восстановить и дать следующий такт.
3. Дождаться, пока опубликованный мостом ответ появится в диалоге, и выполнить опрос входящих.
4. Обработать ответ-письмо пользователя, затем повторно опросить ящик/перезапустить сервис с тем же
   ответом в ящике.

## Ожидаемый результат
- После перезапуска/повторного опроса ранее доставленные сообщения не приходят повторно (0 дублей).
- При сбое отправки ИЛИ записи связки/курсора курсор не продвинут; на следующем такте сообщения
  доставлены хотя бы один раз без потери (at-least-once; повторная доставка приемлема, потеря — нет).
- Собственное эхо (ответ, опубликованный мостом) не доставляется письмом (анти-петля).
- Уже обработанный ответ-письмо не публикуется в Telegram повторно при повторном опросе/перезапуске
  (потреблён единожды).

## Постусловия
Состояние (курсоры + журнал связки) целостно; терминальные исходы (dropped/delivered) не порождают
повторной доставки.

- Концептуальная модель: -> cdm-bridge

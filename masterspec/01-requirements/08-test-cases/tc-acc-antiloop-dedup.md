# Приёмочный тест: анти-петля, дедуп и состояние через перезапуск

## Проверяет
- Функции: -> fn-dm-batch-to-email, -> fn-email-reply-to-tg
- AC: AC-03 (dm), AC-04 (dm), AC-03 (reply)
<!-- Fold guardrail exceptions-first: терминальные состояния помечены, из терминала нет неявных выходов. -->

## Предусловия
Мост включён; есть история доставленных сообщений и запись связки диалога D.

## Входные данные
- Уже доставленные ранее сообщения диалога D.
- Смоделированный сбой отправки письма на одном такте.
- Ответ пользователя, опубликованный мостом в Telegram (собственное эхо).

## Шаги
1. Перезапустить сервис и выполнить повторный опрос по всем диалогам.
2. На одном такте смоделировать неуспешную отправку письма, затем восстановить и дать следующий такт.
3. Дождаться, пока опубликованный мостом ответ появится в диалоге, и выполнить опрос входящих.

## Ожидаемый результат
- После перезапуска/повторного опроса ранее доставленные сообщения не приходят повторно (0 дублей).
- При сбое отправки курсор не продвинут; на следующем такте сообщения доставлены ровно один раз
  (без дубля и без потери).
- Собственное эхо (ответ, опубликованный мостом) не доставляется письмом (анти-петля).

## Постусловия
Состояние (курсоры + журнал связки) целостно; терминальные исходы (dropped/delivered) не порождают
повторной доставки.

## Связи
- Функции: -> fn-dm-batch-to-email, -> fn-email-reply-to-tg
- Концептуальная модель: -> cdm-bridge

# Приёмочный тест: анти-петля, дедуп и состояние через перезапуск

## Проверяет
- Функции: -> fn-dm-batch-to-email, -> fn-email-reply-to-tg
- AC: AC-03 (dm), AC-04 (dm), AC-03 (reply), AC-05 (reply)
- Грани: -> fn-dm-batch-to-email/OE-RESILIENCE, -> fn-dm-batch-to-email/OE-EVIDENCE,
  -> fn-email-reply-to-tg/OE-RESILIENCE, -> fn-email-reply-to-tg/OE-EVIDENCE

## Предусловия
Мост включён; есть история доставленных сообщений и запись связки диалога D.

## Шаги выполнения
1. **Действие:** Перезапустить сервис и выполнить повторный опрос по всем диалогам, содержащим уже
   доставленные ранее сообщения диалога D.
   **Тестовые данные:** рестарт процесса; диалог D с историей доставленных сообщений
   **Ожидаемый результат:**
   - После перезапуска/повторного опроса ранее доставленные сообщения диалога D не приходят повторно
     (0 дублей).

2. **Действие:** На одном такте смоделировать неуспешную отправку письма (SMTP-сбой); дать такту
   завершиться; затем отдельно смоделировать неуспешную запись связки/курсора при успешной
   отправке письма; восстановить и дать следующий такт в обоих случаях.
   **Тестовые данные:** вариант A — инъекция ошибки при отправке письма; вариант B — инъекция ошибки
   при фиксации записи связки/курсора после успешной отправки
   **Ожидаемый результат:**
   - В обоих вариантах курсор диалога не продвинут на неуспешном такте.
   - На следующем такте сообщения доставлены хотя бы один раз без потери (at-least-once; повторная
     доставка приемлема, потеря — нет).

3. **Действие:** Дождаться, пока ответ, опубликованный мостом от лица пользователя, появится в
   диалоге D в Telegram (собственное эхо), затем выполнить опрос входящих личных сообщений.
   **Тестовые данные:** сообщение diалога D, опубликованное мостом (posted_echo)
   **Ожидаемый результат:**
   - Собственное эхо (ответ, опубликованный мостом) не доставляется письмом (анти-петля).

4. **Действие:** Обработать ответ-письмо пользователя на bridged-письмо диалога D (публикация в
   Telegram), затем повторно опросить ящик моста B / перезапустить сервис с тем же ответом-письмом
   всё ещё физически в ящике.
   **Тестовые данные:** ответ-письмо от U на bridged-письмо D; повторный опрос/рестарт после
   обработки
   **Ожидаемый результат:**
   - Уже обработанный ответ-письмо не публикуется в Telegram повторно при повторном опросе/
     перезапуске (потреблён единожды).

## Постусловия
Состояние (курсоры + журнал связки) целостно; терминальные исходы (dropped/delivered) не порождают
повторной доставки.

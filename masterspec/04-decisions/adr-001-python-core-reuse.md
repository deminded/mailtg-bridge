# ADR-001: Реализация на существующем обкатанном ядре (язык — Python)

## Контекст
Мосту нужны зрелый пользовательский клиент Telegram (MTProto) на чувствительной сессии полного
доступа и обкатанный контур почты (приём/отправка, дедуп/связка). У владельца уже есть работающее
питоновское ядро: `channel-reader` (Telethon-поллинг пользовательской сессии) и `de-agent-commons`
(контур почты IMAP/SMTP, журнал связки/дедупа). Нагрузка I/O-bound.
Решение согласовано владельцем 12.07 (adr-кандидат из тикетов).

## Решение
Реализовать мост переиспользованием существующего питоновского ядра: Telethon для пользовательской
сессии Telegram, обкатанная почтовая библиотека для IMAP/SMTP, готовый механизм журнала связки/дедупа.

## Альтернативы
| Альтернатива | Плюсы | Минусы | Почему отвергнута |
|---|---|---|---|
| Rust (grammers) | Скорость, статическая типизация | Незрелость на user-сессии, нет обкатанного ядра, дороже передача | Скорость не включается на I/O-bound; риск на чувствительной сессии |
| Новое ядро с нуля | Чистый дизайн | Долго, теряется обкатанность, повторение уже решённых граблей | Нет выгоды против переиспользования проверенного |

## Последствия
- Слой требований остаётся технологически нейтральным; конкретный стек фиксируется здесь как
  принятое решение и материализуется в слоях спецификаций/кода.
- Гладкая передача и опенсорс-доступность обеспечены зрелостью экосистемы.
- Ограничение: мост наследует эксплуатационные особенности пользовательского клиента Telegram
  (см. -> adr-002-telethon-hybrid-auth, -> rules-security).

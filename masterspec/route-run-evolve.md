## Импакт-анализ по тикетам

### TKT-11 — двухъящичная модель (ящик моста B ≠ адрес пользователя U)
Разводит смешанный «целевой ящик» на B (мост владеет: SMTP-отправка + IMAP-опрос) и U (адрес
доставки + ожидаемый отправитель ответов). Auth: мост действует по ответу/команде ТОЛЬКО если
отправитель=U, письмо пришло на B, in-reply-to доставленного.
- cdm-bridge: +сущности «Ящик моста (B)», «Адрес пользователя (U)»; «Конфигурация» пересобрана
  (креды B + адрес U вместо «целевой ящик»/«доверенные адреса»); +атрибуты B/U; +R-09/R-10
  (composition Config∋B, Config∋U); +2 узла в ER; +инвариант (B→U доставка, приём только от U на B);
  «Мостовое письмо» — отправка с B на U.
- rules-control: единый предикат доверия (ответ И команда: отправитель=U ∧ пришло на B ∧
  in-reply-to доставленного); заголовок расширен; +ссылка на fn-email-reply-to-tg.
- fn-email-reply-to-tg: опрос ящика B; шаг проверки доверия (отправитель=U, in-reply-to); +AC-04
  (чужой адрес/не B → игнор+лог); +ссылка -> rules-control.
- fn-bridge-control-by-email: команда приходит на B, отправитель=U; триггер/предусловия/поток/AC-03.
- nfr-deployability: NFR-DEPLOY-04 — README заводит ящик B (app-пароль); пример конфига покрывает
  креды B (IMAP/SMTP) + адрес U (100% обяз. параметров).
- as-mailtg-bridge: «Внешние взаимодействия» разведены B/U; +ограничение (нужен отдельный аккаунт B).
- context-mailtg-bridge: два почтовых конца (B/U); потоки вход/исход/управление; ER-диаграмма (B→U,
  U→B, опрос B).
- fn-dm-batch-to-email, fn-channel-update-to-email: шаг отправки уточнён «на адрес U через ящик B»
  (снят смешанный «целевой ящик» на несущем действии доставки).
- glossary: +«Ящик моста (B)», +«Адрес пользователя (U)»; «Доверенный адрес» = адрес U; «Bridged-письмо»
  отправляется с B на U.

### TKT-10 — представление сообщения близко к клиенту
Каноническое правило представления (автор+@username, локализ. время, форматирование, цитата-ответ,
медиа-инлайн; HTML) — единый источник в rules-delivery; рендер-функции получают AC.
- rules-delivery: +бизнес-правило представления-фиделити; +инж-правило (HTML/деградация форматирования).
- cdm-bridge / Сообщение Telegram: атрибуты обогащены (отправитель = имя+@username; текст с
  форматированием; ссылка на сообщение-цитату — новая доменная концепция «цитата-ответ»).
- fn-dm-batch-to-email, fn-channel-update-to-email: шаг формирования + AC-06 (опознаваемость как в клиенте).
- glossary: +«Цитата-ответ (quoted-reply)».

### TKT-9 — глубокая ссылка (deep-link) на сообщение
Каноническое правило ссылки — в rules-delivery; рендер-функции получают шаг+AC. Данных не добавляет:
chat-id (Диалог) и msg-id (Сообщение) уже в cdm — ссылка выводима на формировании письма.
- rules-delivery: +бизнес-правило (каждое сообщение несёт deep-link); +инж-правило (формат по типу
  источника: `t.me/c/<chat>/<msg>` приват / `t.me/<user|channel>/<msg>` публ / `tg://` резерв; топик).
- fn-dm-batch-to-email (приват t.me/c/), fn-channel-update-to-email (публ/приват/топик): шаг + AC-05.
- as-mailtg-bridge: +возможность (deep-link на оригинал).
- glossary: +«Глубокая ссылка (deep-link)».

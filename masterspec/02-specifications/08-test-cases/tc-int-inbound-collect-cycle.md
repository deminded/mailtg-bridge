# Интеграционный тест: такт сбора входящих (гейт + батч + медиа + деградация + дедуп)

## Проверяет
- Сценарий: -> scn-inbound-collect-cycle
- Функция: -> fn-dm-batch-to-email, -> fn-channel-update-to-email, -> fn-media-in-email
- Путь: основной + ветвления «личка/канал-гейт», «медиа», «деградация размера»; ошибки «сбой отправки», «FloodWait»

## Предусловия
Мост включён; сессия действительна. Заглушки: -> cmp-tg-gateway отдаёт заранее заданные сообщения по
диалогам (личка D1 с N>1 сообщениями и изображением; канал C1 в белом списке; канал C2 вне списка без
упоминания; канал C3 вне списка с упоминанием; сообщение-эхо в D1); -> cmp-email-out подключён к
SMTP-заглушке ящика B, фиксирующей отправленные письма; лимит размера письма задан так, что один набор
вложений его превышает; курсоры диалогов на исходной позиции.

## Входные данные
Пакет сообщений за такт: D1 {m1, m2(изображение), m3(эхо), m4(вложение > порога)}, C1 {c11}, C2 {c21},
C3 {c31(упоминание)}.

## Шаги
1. Запустить такт сбора. — компонент: -> cmp-bridge-orchestrator/cap-run-inbound-cycle
2. Выбрать сообщения после курсора и отсеять эхо m3. — -> cmp-tg-gateway/cap-fetch-since-cursor, -> cmp-tg-gateway/cap-detect-own-echo
3. Применить гейт: D1 — пропуск; C1 — пропуск (белый список); C2 — отсев; C3 — пропуск (упоминание). — -> cmp-tg-gateway/cap-apply-addressing-gate
4. Собрать батчи по диалогу и сформировать письма с медиа и деградацией. — -> cmp-bridge-orchestrator/cap-assemble-dialog-batch, -> cmp-email-out/cap-compose-batch-email, -> cmp-email-out/cap-render-media, -> cmp-email-out/cap-degrade-on-oversize
5. Отправить письма на U и продвинуть курсоры. — -> cmp-email-out/cap-send-from-b, -> cmp-state-store/cap-manage-ledger, -> cmp-state-store/cap-manage-cursor
6. Смоделировать сбой отправки на повторном прогоне для D1, затем восстановление и новый такт. — -> cmp-email-out/cap-send-from-b, -> cmp-bridge-orchestrator/cap-apply-backoff

## Ожидаемые эффекты
- На ящик B (заглушка) уходит по одному письму на D1, C1, C3; письма для C2 нет.
- Письмо D1 содержит m1, m2, m4 в хронологическом порядке; m3 (эхо) отсутствует; изображение m2 — инлайн;
  вложение m4 — текстовое указание (> порога); при превышении лимита письмо раздроблено, порядок сохранён.
- Записи связки созданы для доставленных диалогов; курсоры продвинуты только по успешно отправленным.
- При смоделированном сбое отправки D1 курсор D1 не продвинут; после восстановления сообщения доставлены
  без дубля для уже доставленных и без потери недоставленных.

## Проверяемые условия
- Число писем за такт = 3 (D1, C1, C3); C2 не порождает письма (гейт).
- В письме D1 порядок сообщений хронологический; эхо исключено; изображение инлайн; крупное вложение — указание.
- Каждое итоговое письмо ≤ лимита размера (деградация/дробление сработали).
- Повторный такт/перезапуск не создаёт дублей доставленных сообщений (0 дублей).
- При сбое отправки курсор не продвинут (at-least-once).

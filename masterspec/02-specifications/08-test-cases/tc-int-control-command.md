# Интеграционный тест: управление мостом письмом (доверие + токен + идемпотентность)

## Проверяет
- Сценарий: -> scn-control-command
- Функция: -> fn-bridge-control-by-email
- Путь: основной + ветвления «недоверенный отправитель», «неверный/отсутствующий токен», «идемпотентная команда»

## Предусловия
Сконфигурирован адрес U и секрет-токен темы. Мост в состоянии «включён». Заглушки: IMAP-заглушка ящика B
с письмами-командами; SMTP-заглушка фиксирует подтверждения; -> cmp-state-store хранит состояние моста.

## Входные данные
Письма-команды на B: K1 {from=U, «выключить», токен верный, in_reply_to доставленного}; K2 {from=чужой,
«включить», токен верный}; K3 {from=U, «включить», токен отсутствует/неверный}; K4 {from=U, «выключить»
повторно, токен верный} — на уже выключенном мосте.

## Шаги
1. Запустить такт опроса ящика. — компонент: -> cmp-bridge-orchestrator/cap-run-mailbox-cycle
2. Проверить доверие и токен, классифицировать команды. — -> cmp-email-in/cap-authenticate-sender, -> cmp-email-in/cap-classify-message
3. Применить K1 (выключить) к состоянию и подтвердить. — -> cmp-state-store/cap-manage-bridge-state, -> cmp-email-out/cap-send-notice
4. Обработать K2, K3, K4. — -> cmp-email-in/cap-authenticate-sender, -> cmp-state-store/cap-manage-bridge-state
5. Перезапустить сервис и прочитать состояние. — -> cmp-state-store/cap-manage-bridge-state

## Ожидаемые эффекты
- K1 → мост выключен, подтверждение на U отправлено.
- K2 (чужой адрес) → состояние не изменено, факт в логе, подтверждения нет.
- K3 (нет/неверен токен) → состояние не изменено, факт в логе, подтверждения нет.
- K4 (выключить на выключенном) → состояние не меняется, подтверждение идемпотентно.
- После перезапуска состояние «выключен» сохранено.

## Проверяемые условия
- Итоговое состояние моста = «выключен».
- Подтверждения отправлены только на доверенные исполненные команды (K1; K4 — идемпотентное подтверждение).
- Недоверенная (K2) и непрошедшая токен (K3) команды не меняют состояние и залогированы.
- Состояние переживает перезапуск.

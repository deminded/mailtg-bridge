# Интеграционный тест: первичная авторизация Telegram (интерактивный setup → сессия)

## Проверяет
- Сценарий: -> scn-first-run-setup
- Функция: -> fn-first-run-setup
- Путь: основной (телефон/код/2FA → файл сессии) + ветвление «сессия уже существует»; ошибки «Telegram недоступен при авторизации», «неверный код/пароль»
- Грани живой эксплуатации: -> fn-first-run-setup/OE-EVIDENCE, -> fn-first-run-setup/OE-SECURITY, -> fn-first-run-setup/OE-RESILIENCE
- Строки каталога: -> tc-flt-first-run-setup/FLT-000, -> tc-flt-first-run-setup/FLT-001, -> tc-flt-first-run-setup/FLT-002

## Fidelity эксплуатационной проверки
- Источник OE-маппинга: -> scn-first-run-setup § «Реализация контракта живой эксплуатации»
- Профиль / fixture / внешняя среда: заглушка MTProto авторизации (-> api-telegram-userclient) с управляемыми ответами success/unavailable/bad-code/bad-password; интерактивный ввод подан fixture-значениями (не реальный терминал)
- Fidelity: production-like
- Что реально проверяется: создание файла сессии с правами 600, headless-старт при уже действительной сессии, отказ авторизации при недоступности Telegram и при неверном коде/пароле
- Residual risk: реальный интерактивный терминал (эхо пароля, локаль ввода) проверяется отдельно live-e2e (-> tc-acc-deploy-and-security)

## Предварительные действия
- Действительной сессии нет; задана конфигурация доступа Telegram и путь к файлу сессии; у оператора есть валидные телефон/код/2FA-пароль для happy-path.

## Шаги выполнения
1. **Действие:** Запустите интерактивную команду инициализации с валидными телефоном, кодом и 2FA-паролем.
   **Тестовые данные:** phone=+70000000000; code=12345; password=valid-2fa
   **Ожидаемый результат:**
   - Создан файл сессии с правами доступа только для владельца (600), вне репозитория; состояние сессии — «действительна», notified=false.

2. **Действие:** Запустите фоновый сервис при уже существующей действительной сессии.
   **Тестовые данные:** N/A — сессия из шага 1 используется как есть
   **Ожидаемый результат:**
   - Сервис стартует headless, без запроса кода; интерактивный шаг не выполняется.

3. **Действие:** Сбросьте сессию и запустите интерактивную инициализацию при недоступном Telegram (transient на этапе запроса кода).
   **Тестовые данные:** phone=+70000000000; telegram-auth-mode=unavailable
   **Ожидаемый результат:**
   - Авторизация не завершается; файл сессии не создаётся; явная ошибка и предложение повторить.

4. **Действие:** Повторите интерактивную инициализацию с неверным кодом подтверждения.
   **Тестовые данные:** phone=+70000000000; code=00000 (неверный); telegram-auth-mode=bad-code
   **Ожидаемый результат:**
   - Авторизация не завершается; файл сессии не создаётся; явная ошибка и предложение повторить ввод.

5. **Действие:** Повторите интерактивную инициализацию с верным кодом, но неверным 2FA-паролем.
   **Тестовые данные:** phone=+70000000000; code=12345; password=wrong-2fa; telegram-auth-mode=bad-password
   **Ожидаемый результат:**
   - Авторизация не завершается; файл сессии не создаётся; явная ошибка и предложение повторить ввод.

6. **Действие:** Проверьте диагностический журнал по идентификатору операции и окну времени для отказов шагов 3–5.
   **Тестовые данные:** correlation-id=int-setup-001; time-window=5 минут; expected-event=auth-failed
   **Ожидаемый результат:**
   - Найдены ровно 3 записи `auth-failed` с причинами `unavailable`, `bad-code`, `bad-password` соответственно; файл сессии в каталоге состояния отсутствует после каждой из них.

## Проверяемые условия
- Файл сессии создаётся только при успешной авторизации (шаг 1), с правами 600.
- Действительная сессия даёт headless-старт без интерактивного шага (шаг 2).
- Ни один из отказов (недоступность, неверный код, неверный пароль) не создаёт файл сессии и не оставляет сервис в цикле повторного запроса без явной ошибки.

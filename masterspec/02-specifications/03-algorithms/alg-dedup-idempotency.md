# Алгоритм: дедуп доставки, анти-петля, идемпотентность приёма

## Контекст применения
Сквозной алгоритм целостности: применяется в -> scn-inbound-collect-cycle (дедуп доставки, анти-петля) и
-> scn-outbound-reply / -> scn-control-command (однократное потребление). Опирается на курсор диалога и
журнал связки из -> cmp-state-store. Реализует -> rules-integrity.

## Вход
Идентичность сообщения (диалог + идентификатор сообщения в диалоге); признак собственного эха; курсор
диалога; журнал связки; множество потреблённых писем; идентификатор входящего письма (In-Reply-To / UID).

## Выход
Решения: «доставлять / уже доставлено», «эхо → отсеять», «публиковать ответ / уже потреблён».

## Правила
1. Дедуп доставки: сообщение обрабатывается, только если его идентификатор в диалоге строго больше курсора.
   Иначе (≤ курсора) → уже доставлено, повторно не доставляется (идемпотентно к повторному опросу/перезапуску).
   Курсор — монотонный high-watermark (-> alg-batch-per-dialog-cycle): отсутствующие/удалённые id (сообщение
   удалено до обработки, дыра в нумерации) толерируются и НЕ блокируют доставку последующих; непрерывность id
   не требуется.
2. Анти-петля: если сообщение распознано как собственное эхо моста (опубликовано мостом от лица
   пользователя) → отсеять, письма нет. Иначе → продолжить обработку. Метод детекции — по идентичности:
   сообщение помечено исходящим/авторства пользователя И его идентификатор присутствует в наборе
   опубликованных мостом (ведётся при -> cmp-tg-gateway/cap-post-as-user). Набор ограничен (см. инварианты).
3. At-least-once на сбое записи: если письмо отправлено, но запись связки/продвижение курсора не удались →
   курсор не продвигается; на следующем такте сообщение пересобирается и доставляется повторно (дубль
   приемлем). Иначе (всё удалось) → курсор продвинут, повторной доставки нет. Потеря сообщения не допускается.
4. Идемпотентность приёма (ответ/команда): входящее письмо обрабатывается, только если его идентификатор
   отсутствует во множестве потреблённых. Иначе → пропустить (уже обработано, в Telegram/состояние повторно
   не применяется). Отметка «потреблено» ставится ТОЛЬКО после успешной публикации/исполнения (правило 5).
5. Порядок фиксации приёма: сначала действие (публикация ответа / применение команды), затем отметка
   «потреблено». Если действие не удалось → отметка не ставится, повтор на следующем такте (симметрично правилу 3).
6. Маршрутизация ответа: ключ связки — диалог, не отдельное сообщение; ответ без записи связки → игнор
   (не публикуется). Новый (иной) ответ на тот же диалог → публикуется дополнительно (дедупа по содержанию нет).

## Порядок проверок
Дедуп по курсору (правило 1) → анти-петля (правило 2) — на входящем потоке; на приёме: проверка
«потреблено» (правило 4) → действие → отметка (правило 5). Отметка/продвижение — всегда ПОСЛЕ действия.

## Причины выбора порядка
«Идемпотентность важнее скорости: любой такт можно повторить без побочных эффектов доставки» — -> rules-integrity.

## Инварианты
- На одно мостовое письмо — ровно одна запись связки; ключ — диалог.
- Множества «опубликовано мостом» и «потреблённые письма» ограничены и вычищаются согласованно с retention
  журнала связки (-> alg не задаёт число; граница — конфигурация, см. -> data-bridge-store).
- Дубль доставки допустим, потеря сообщения и повторная публикация уже потреблённого — нет.

# Алгоритм: отступ при FloodWait и преходящих ошибках

## Контекст применения
Применяется в -> cmp-bridge-orchestrator/cap-apply-backoff при ошибках внешних сторон в любом такте
(-> scn-inbound-collect-cycle, -> scn-outbound-reply, -> scn-control-command, -> scn-session-invalid-alert).
Реализует требование устойчивости -> rules-integrity и -> nfr-operability (NFR-OPS-06).

## Вход
Класс ошибки внешней стороны от -> cmp-tg-gateway/cap-surface-session-errors или почтового контракта:
FloodWait (с указанным Telegram интервалом ожидания) / преходящая (сеть, временный отказ IMAP/SMTP) /
сессия недействительна-отозвана.

## Выход
Решение: «выдержать отступ и продолжить на следующем такте» либо «остановить опрос (блокирующее)».

## Правила
1. Если ошибка — FloodWait → не повторять раньше указанного Telegram интервала ожидания; текущий такт для
   затронутого источника пропускается, состояние (курсор) не меняется. Иначе → перейти к правилу 2.
2. Если ошибка — преходящая → отступить не короче заданного минимального интервала (нижняя граница
   отступа задаётся конфигурацией; допускается нарастающий отступ при повторах), работу перенести на
   следующий такт. Иначе → перейти к правилу 3.
3. Если ошибка — сессия недействительна/отозвана → НЕ ретраить: перейти в -> scn-session-invalid-alert
   (блокирующее состояние, разовое уведомление). Иначе (не один из трёх классов) → трактовать как
   преходящую (правило 2) — безопасный дефолт «отступить и продолжить», не срывать и не терять.
4. Между повторами не входить в плотный цикл: интервал между попытками ≥ заданного отступа (нет busy-loop).
5. На любом отступе сообщения и письма не теряются: непродвинутый курсор и непотреблённое письмо
   гарантируют повторную обработку (at-least-once, -> alg-dedup-idempotency).

## Порядок проверок
Сначала FloodWait (уважить директиву Telegram по времени), затем преходящая, затем блокирующая сессия;
неизвестный класс → как преходящая (без потери).

## Причины выбора порядка
FloodWait несёт явный интервал ожидания от Telegram и приоритетен; блокирующая сессия не ретраится, чтобы
не зациклиться (-> nfr-operability NFR-OPS-06).

## Инварианты
- Интервал между повторами ≥ заданного отступа (отсутствие busy-loop).
- Потерянных сообщений = 0 при инъекции FloodWait/сетевых ошибок.
- Блокирующее состояние сессии не порождает повторных уведомлений (разовое).

# Алгоритм: батчинг «диалог за такт» и дисциплина курсора

## Контекст применения
Применяется в -> cmp-bridge-orchestrator/cap-assemble-dialog-batch и во взаимодействии с
-> cmp-state-store/cap-manage-cursor на шагах 6 и 12 сценария -> scn-inbound-collect-cycle. Реализует
единицу доставки «одно письмо на диалог за такт» и монотонное продвижение курсора (-> rules-delivery, -> rules-integrity).

## Вход
Прошедшие гейт сообщения текущего такта, сгруппированные по диалогу; текущие курсоры диалогов; признак
успеха отправки письма и создания записи связки (по итогам шага отправки).

## Выход
Не более одного батча (упорядоченного письма) на диалог за такт; продвинутый ЛИБО непродвинутый курсор диалога.

## Правила
1. Сгруппировать прошедшие гейт сообщения по диалогу; топик группы — отдельный диалог (батчинг по топику).
2. Если у диалога есть ≥1 прошедшее гейт сообщение → сформировать один батч в хронологическом порядке.
   Иначе → батч для диалога не создаётся (письма нет).
3. Каждое адресованное сообщение входит ровно в один батч своего диалога за такт (не дублируется между батчами).
4. Отправку письма батча выполняет -> cmp-email-out; продвижение курсора зависит от её исхода (правило 5).
5. Порядок фиксации диалога СТРОГО: (1) отправить письмо-батч (-> cmp-email-out) → (2) записать связку
   (ledger) → (3) продвинуть курсор. Продвижение — до МАКСИМАЛЬНОГО msg_id, полученного из fetch в этом
   такте (курсор = монотонный high-watermark по выбранным id), а НЕ только до последнего сообщения батча.
   Реакции на сбой: сбой отправки → связку не писать, курсор не двигать (пересбор на следующем такте);
   отправка удалась, но запись связки НЕ удалась → курсор НЕ продвигать (возможна повторная доставка —
   приемлемый дубль, снимается дедупом -> alg-dedup-idempotency); краш между отправкой и продвижением
   курсора → не более одной повторной доставки на следующем такте (at-least-once). Потеря не допускается.
6. Курсор изменяется только в сторону возрастания (монотонность); повторный такт с непродвинутым курсором
   пересобирает те же сообщения идемпотентно.
7. Пропуски и удалённые id НЕ стопорят прогресс: отсутствующий/удалённый id (сообщение удалено до
   обработки, дыра в нумерации, сообщение отсеяно гейтом/эхом) перешагивается high-watermark'ом и НИКОГДА
   не блокирует доставку последующих сообщений; непрерывность потока id не требуется и не проверяется.

## Порядок проверок
Сначала группировка и формирование батча, затем отправка, только затем — продвижение курсора. Продвижение
курсора строго ПОСЛЕ подтверждённой отправки — иначе теряются недоставленные сообщения. Порядок фиксации
внутри диалога: отправка → запись связки → продвижение курсора (правило 5).

## Причины выбора порядка
«Лучше пропустить продвижение курсора и пересобрать, чем потерять сообщение» — -> rules-integrity.

## Инварианты
- ≤ 1 батч на диалог за такт; ≤ 1 письмо на батч (1:1).
- Порядок сообщений в батче — хронологический и сохраняется (в т.ч. при дроблении в -> alg-oversize-degrade).
- Курсор монотонно не убывает; продвигается до макс. выбранного в такте msg_id (high-watermark) и НЕ
  опережает ни одно прошедшее гейт сообщение, не вошедшее в успешно доставленный батч. Отсеянные
  гейтом/эхом и удалённые/отсутствующие id перешагиваются (прогресс не стопорится).

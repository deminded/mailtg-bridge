# Алгоритм: деградация при превышении лимита размера письма

## Контекст применения
Применяется в -> cmp-email-out/cap-degrade-on-oversize на шаге 10 сценария -> scn-inbound-collect-cycle
при формировании письма-батча. Реализует -> rules-delivery (деградация, а не молчаливый срыв) и закрывает
свёрнутый дефект CRIT-004/MAJOR-005.

## Вход
Черновик письма (тело + встроенные изображения + вложения-файлы + указания); порог размера вложений и
лимит размера письма (провайдера) из конфигурации (-> data-bridge-store).

## Выход
Одно ИЛИ несколько писем, каждое в пределах лимита размера; ничего не потеряно молча.

## Правила
1. Оценить итоговый размер письма. Если ≤ лимита → отправить как одно письмо (деградация не нужна). Иначе →
   перейти к правилу 2.
2. Понизить крупные не-изображения (файлы > порога и/или наиболее весомые вложения) до текстового указания
   (имя/размер/тип) без тела. Пересчитать размер. Если ≤ лимита → отправить одно письмо. Иначе → перейти к правилу 3.
3. Раздробить батч на несколько писем так, чтобы каждое было ≤ лимита, сохраняя хронологический порядок
   сообщений (разрез между сообщениями, не внутри сообщения). Если даже одно сообщение с инлайн-изображением
   превышает лимит → понизить это изображение до текстового указания (имя/тип) и повторить оценку. Иначе
   (после дробления все письма ≤ лимита) → отправить их по порядку.
4. Инвариант «ничего молча»: любой элемент, не попавший в тело (понижен/опущен), обозначается текстовым
   указанием в письме; сорванной или молча потерянной доставки быть не должно.

## Порядок проверок
Порядок деградации фиксирован и щадящий: сначала понижение крупных вложений (дешевле для обзорности), затем
дробление батча, и лишь в крайнем случае — понижение инлайн-изображения. Порог вложений применяется до
оценки лимита письма (изображение остаётся инлайн независимо от порога вложений — см. -> dict-media-disposition).

## Причины выбора порядка
«Крупные бинарные данные не раздувают ящик; приоритет — обозримость почты» — -> rules-delivery.

## Инварианты
- Каждое итоговое письмо ≤ лимита размера провайдера.
- Хронологический порядок сообщений сохранён при дроблении.
- Каждое непереданное телом вложение имеет текстовое указание (нет молчаливой потери).

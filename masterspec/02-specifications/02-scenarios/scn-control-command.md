# Сценарий: Управление мостом письмом (включить/выключить)

## Цель сценария
Реализует смену состояния моста (вкл/выкл) командой письмом с доверенного адреса и подтверждение. -> fn-bridge-control-by-email

## Триггер
Наступил такт опроса ящика B; обнаружено письмо-команда (служебная тема).

## Предусловия
В конфигурации заданы адрес U и, опционально, секрет-токен темы; команды принимаются на ящике B.
(Команды исполняются и при «выключено» — иначе мост не включить: гейт «включён» на команды не распространяется.)

## Последовательность шагов
1. Выбрать новые письма на ящике B. -> cmp-email-in/cap-poll-b-imap · -> api-mailbox-imap-smtp
2. Проверить предикат доверия (отправитель = U, пришло на B, in-reply-to доставленного) И секрет-токен, если задан. -> cmp-email-in/cap-authenticate-sender
3. Классифицировать письмо как команду «включить»/«выключить». -> cmp-email-in/cap-classify-message
4. Применить команду к состоянию моста (идемпотентно). -> cmp-state-store/cap-manage-bridge-state
5. Отправить подтверждение письмом на U. -> cmp-email-out/cap-send-notice · -> api-mailbox-imap-smtp
6. Отметить письмо-команду потреблённым. -> cmp-email-in/cap-mark-consumed, -> cmp-state-store/cap-manage-consume-markers

## Ветвления
- Предикат доверия не выполнен (шаг 2: чужой адрес / не на B / не in-reply-to) → команда игнорируется,
  состояние не меняется, факт в лог, подтверждение НЕ отправляется; иначе → продолжить.
- Задан секрет-токен, а он отсутствует/неверен (шаг 2) → игнор + лог, подтверждения нет; иначе → продолжить.
- Команда «включить» на включённом / «выключить» на выключенном (шаг 4) → состояние не меняется,
  подтверждение идемпотентно; иначе → смена состояния + подтверждение.
- Письмо не команда, а ответ на bridged-письмо (шаг 3) → передать в -> scn-outbound-reply.
- Нераспознанная команда (вне вкл/выкл версии 1) → не исполняется, состояние неизменно.

## Ошибки и таймауты
- Преходящая ошибка IMAP/SMTP → такт пропускается, письмо-команда остаётся на B, обработка на следующем
  такте; отметка «потреблено» ставится только после успешного применения. -> cmp-bridge-orchestrator/cap-apply-backoff

## Постусловия
Состояние моста переведено в требуемое (или подтверждено идемпотентно) и переживает перезапуск;
подтверждение отправлено; недоверенная/непрошедшая токен команда не изменила состояние и залогирована.

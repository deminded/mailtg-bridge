# Сценарий: Недействительная сессия — остановка и уведомление

## Цель сценария
Реализует безопасную реакцию на отзыв/недействительность сессии в ходе работы: остановить опрос, один раз
уведомить пользователя U, не зациклиться на той же ошибке (блокирующее состояние). -> fn-first-run-setup (AC-03), -> cdm-bridge

## Триггер
Telegram отклонил сессию (отозвана/недействительна) при выборке, скачивании или публикации.

## Предусловия
Сервис работал со состоянием сессии «действительна».

## Последовательность шагов
1. Распознать ошибку как класс «сессия недействительна/отозвана». -> cmp-tg-gateway/cap-surface-session-errors · -> api-telegram-userclient
2. Установить состояние сессии «недействительна» (блокирующее). -> cmp-state-store/cap-manage-session-health
3. Остановить дальнейший опрос Telegram в текущем и последующих тактах до реинициализации. -> cmp-bridge-orchestrator/cap-enforce-delivery-gates
4. Отправить однократное письмо-уведомление на U о необходимости повторной инициализации. -> cmp-email-out/cap-send-notice · -> api-mailbox-imap-smtp

## Ветвления
- Состояние сессии уже «недействительна» (наступил следующий такт) → опрос НЕ идёт, повторное уведомление
  НЕ отправляется (нет зацикливания); иначе (первый переход) → шаги 2–4.
- Выполнена повторная интерактивная инициализация → -> scn-first-run-setup возвращает сессию в
  «действительна» И сбрасывает notified=false (следующий отзыв сессии снова уведомит U); сервис
  возобновляет работу.

## Ошибки и таймауты
- Отправка письма-уведомления не удалась (шаг 4) → повтор уведомления на следующем такте допускается
  ТОЛЬКО для неотправленного уведомления (флаг «уведомлено» ставится по факту отправки), без busy-loop;
  состояние остаётся блокирующим. -> cmp-bridge-orchestrator/cap-apply-backoff

## Постусловия
Опрос Telegram остановлен; пользователь U уведомлён единожды; сервис в блокирующем состоянии до повторной
инициализации; повторного зацикливания на той же ошибке нет.

# Компонент: Исходящая почта (формирование и отправка с ящика B)

## Назначение
Превращает батч диалога в одно письмо, максимально близкое к виду в клиенте Telegram (автор,
локализованное время, форматирование, цитата-ответ, глубокая ссылка), встраивает изображения инлайн,
раскладывает прочие вложения по порогу, деградирует при превышении лимита размера письма и отправляет
письмо с ящика моста (B) на адрес пользователя (U) по SMTP. Тот же компонент шлёт подтверждения команд
и уведомление о недействительной сессии.

## Ответственность
- Формирование HTML-тела с обязательным текстовым запасным вариантом из батча (порядок сообщений сохранён).
- Построение глубокой ссылки по типу источника (публичный/приватный/топик; для лички 1:1 — без ссылки).
- Представление медиа: изображение — инлайн; прочее — файлом (≤ порога) или текстовым указанием (> порога).
- Деградация при превышении лимита размера письма: крупные вложения → указание, при остаточном
  превышении — дробление батча на несколько писем с сохранением порядка; без молчаливого срыва.
- Отправка письма с ящика B на адрес U по SMTP с заголовками треда (устойчивый Message-ID + тег источника).
- Отправка подтверждений команд и письма-уведомления о недействительной/отозванной сессии.

## Не делает
- Не скачивает вложения из Telegram (это -> cmp-tg-gateway/cap-download-media) — получает содержимое на вход.
- Не хранит запись связки (это -> cmp-state-store) — но возвращает устойчивый Message-ID для её создания.
- Не решает адресованность и не опрашивает ящик (это -> cmp-tg-gateway, -> cmp-email-in).

## Возможности (capabilities)

### cap-compose-batch-email
- **Что делает:** собирает HTML+текст письмо из батча: на каждое сообщение — автор (имя и @username),
  локализованное время, текст с сохранённым форматированием, показанная цитата-ответ, тег источника.
- **Вход (логически):** батч диалога (упорядоченные сообщения с метаданными), тег источника, локаль/таймзона.
- **Выход (логически):** черновик письма (HTML-тело + текстовый вариант + встроенные изображения) с
  устойчивым Message-ID.
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email

### cap-build-deeplink
- **Что делает:** строит глубокую ссылку на оригинал по типу источника (публичный `t.me/<username>/<msg>`,
  топик `.../<topic>/<msg>`, приватный `t.me/c/<internal>/<msg>`); для лички 1:1 — ссылки нет.
- **Вход (логически):** тип источника, идентификаторы диалога/топика/сообщения, наличие username.
- **Выход (логически):** строка глубокой ссылки либо отсутствие ссылки (личка 1:1).
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email

### cap-render-media
- **Что делает:** выбирает способ представления вложения (инлайн-изображение / приложенный файл /
  текстовое указание) по типу и порогу; при недоступности содержимого — указание о недоступном вложении.
- **Вход (логически):** вложения сообщения (содержимое + имя/размер/тип), порог размера.
- **Выход (логически):** элементы тела письма: инлайн-картинки, файлы-вложения, текстовые указания.
- **Используется функциями АС/ФП:** -> fn-media-in-email

### cap-degrade-on-oversize
- **Что делает:** при риске превышения лимита размера письма понижает крупные вложения до указаний, а
  при остаточном превышении дробит батч на несколько писем, сохраняя порядок; доставка не срывается.
- **Вход (логически):** черновик письма, лимит размера письма из конфигурации.
- **Выход (логически):** одно или несколько писем в пределах лимита.
- **Используется функциями АС/ФП:** -> fn-media-in-email

### cap-send-from-b
- **Что делает:** отправляет письмо с ящика B на адрес U по SMTP с заголовками, пригодными для
  распознавания ответа (Message-ID, тег источника в теме).
- **Вход (логически):** готовое письмо, креды/адрес ящика B, адрес U.
- **Выход (логически):** факт отправки (успех/неуспех) и устойчивый Message-ID отправленного письма.
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email

### cap-send-notice
- **Что делает:** формирует и отправляет служебное письмо на U: подтверждение команды вкл/выкл либо
  уведомление о недействительной сессии (однократно, без зацикливания).
- **Вход (логически):** тип уведомления, повод.
- **Выход (логически):** отправленное служебное письмо.
- **Используется функциями АС/ФП:** -> fn-bridge-control-by-email, -> fn-first-run-setup

## Зависимости
- Ящик моста (B) как SMTP-отправитель (внешняя система) — -> api-mailbox-imap-smtp.
- -> cmp-state-store — порог/лимит/адреса/локаль из конфигурации; Message-ID для записи связки.
- -> cmp-tg-gateway — содержимое вложений.

## Связи
- Реализует функции АС/ФП: -> fn-dm-batch-to-email, -> fn-channel-update-to-email, -> fn-media-in-email, -> fn-bridge-control-by-email, -> fn-first-run-setup
- Сценарии: -> scn-inbound-collect-cycle, -> scn-control-command, -> scn-session-invalid-alert
- Алгоритмы: -> alg-oversize-degrade
- API: -> api-mailbox-imap-smtp
- Данные: -> data-bridge-store

## Evolve-дельта (13.07, живой тест)
**Sent-копия (TKT-14).** После успешного SMTP `SmtpMailer.send` вызывает `_archive_sent(raw)`:
если `save_sent_copy` — открыть IMAP к B (та же security-политика), `APPEND` в `sent_folder` с флагом
`\Seen`, затем logout. Всё в try/except: любой сбой архивации логируется (warning, operation=archive)
и НЕ пробрасывается — доставка уже состоялась. Папка Sent не читается поллером (poll = только INBOX),
петли нет.

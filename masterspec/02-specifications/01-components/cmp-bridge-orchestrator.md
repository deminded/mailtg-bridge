# Компонент: Оркестратор моста (поллинг-цикл)

## Назначение
Единая точка управления тактами моста: по расписанию запускает такт сбора входящих (Telegram → почта)
и такт опроса ящика (почта → Telegram / команды), проводит их через гейты состояния (мост включён,
сессия действительна), собирает адресованные сообщения диалога в один батч и удерживает дисциплину
отступа при сбоях. Однопроцессный поллер (модель конкурентного доступа — -> adr-001-python-core-reuse):
такты не пересекаются, состояние общее. Непересечение ГАРАНТИРУЕТСЯ эксклюзивной блокировкой (flock на
lock-файле): второй/перекрывающий такт не стартует, пока блокировка занята; краш процесса освобождает её.

## Ответственность
- Запуск такта сбора входящих с периодичностью «сбор» и такта опроса ящика с периодичностью
  «отправка/опрос» (независимые интервалы конфигурации).
- Проверка перед доставкой/публикацией: мост включён И сессия действительна; иначе такт не производит
  доставку/публикацию.
- Сборка адресованных сообщений одного диалога за такт в один батч (единица доставки — диалог за такт).
- Управление отступом при FloodWait/преходящих ошибках: пропустить такт, продолжить на следующем, без
  плотного цикла (at-least-once).
- Оркестрация порядка: сбор → гейт → батч → формирование письма → отправка → продвижение курсора;
  при недействительной сессии — остановка опроса и запуск уведомления.
- Гарантия единственного экземпляра: удержание эксклюзивной блокировки (flock на lock-файле) на время
  такта; перекрывающийся или второй запуск (в т.ч. по расписанию cron/таймера ОС) не стартует, пока
  блокировка занята; краш освобождает блокировку — следующий такт продолжает без дубля и без потери
  (-> nfr-operability NFR-OPS-05, -> nfr-deployability NFR-DEPLOY-05).

## Не делает
- Не читает Telegram и не пишет в него напрямую (это -> cmp-tg-gateway).
- Не формирует и не отправляет письма (это -> cmp-email-out), не опрашивает ящик (это -> cmp-email-in).
- Не хранит курсоры/журнал/состояние (это -> cmp-state-store) — только читает и командует.
- Не принимает решение адресованности (гейт) и деградации размера — делегирует владельцам возможностей.

## Возможности (capabilities)

### cap-run-inbound-cycle
- **Что делает:** проводит один такт сбора входящих от триггера расписания до продвижения курсоров.
- **Вход (логически):** сигнал такта сбора; состояние моста и сессии; список отслеживаемых диалогов.
- **Выход (логически):** для каждого диалога с адресованными сообщениями — сформированное и отправленное
  письмо-батч, продвинутый курсор; при сбое — курсор не продвинут.
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email

### cap-run-mailbox-cycle
- **Что делает:** проводит один такт опроса ящика моста: обнаруженные ответы маршрутизирует в Telegram,
  команды исполняет.
- **Вход (логически):** сигнал такта опроса; состояние моста и сессии.
- **Выход (логически):** опубликованные ответы / изменённое состояние моста / проигнорированные письма.
- **Используется функциями АС/ФП:** -> fn-email-reply-to-tg, -> fn-bridge-control-by-email

### cap-enforce-delivery-gates
- **Что делает:** до любой доставки/публикации проверяет два блокирующих условия — мост включён и
  сессия действительна; команды управления пропускаются даже при «выключено» (иначе мост не включить).
- **Вход (логически):** признак «доставка включена», признак «сессия действительна».
- **Выход (логически):** разрешение/запрет производить доставку и публикацию в текущем такте.
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email, -> fn-email-reply-to-tg, -> fn-bridge-control-by-email

### cap-assemble-dialog-batch
- **Что делает:** группирует прошедшие гейт сообщения одного диалога за такт в упорядоченный
  (хронологически) батч — не более одного батча на диалог за такт.
- **Вход (логически):** прошедшие гейт сообщения диалога за такт.
- **Выход (логически):** один батч диалога либо отсутствие батча (нет адресованных сообщений).
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email

### cap-apply-backoff
- **Что делает:** при FloodWait/преходящей ошибке внешней стороны выдерживает отступ и переносит работу
  на следующий такт без busy-loop; при недействительной сессии — не ретраит, а останавливает опрос.
- **Вход (логически):** класс ошибки внешней стороны (FloodWait / преходящая / сессия недействительна).
- **Выход (логически):** решение «отступить и продолжить позже» либо «остановить опрос (блокирующее)».
- **Используется функциями АС/ФП:** -> fn-dm-batch-to-email, -> fn-channel-update-to-email, -> fn-first-run-setup

## Зависимости
- -> cmp-tg-gateway — читать входящие и публиковать ответы через пользовательскую сессию Telegram.
- -> cmp-email-out — формировать и отправлять письмо-батч, подтверждения и уведомления.
- -> cmp-email-in — опрашивать ящик моста на ответы и команды.
- -> cmp-state-store — читать конфигурацию/состояние, продвигать курсоры, вести журнал связки.
- Расписание (планировщик ОС / таймеры) — источник тактов (-> nfr-deployability).

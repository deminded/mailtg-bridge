# Компонент: Входящая почта (опрос ящика B, ответы и команды)

## Назначение
Опрашивает ящик моста (B) по IMAP, отличает ответ на bridged-письмо от команды управления, проверяет
единый предикат доверия (отправитель = адрес U, письмо пришло на B, является ответом на ранее
доставленное; для команды — опциональный секрет-токен) и обеспечивает однократное потребление письма
(идемпотентность приёма к повторному опросу/перезапуску).

## Ответственность
- Выборка новых писем на ящике B (входящие ответы и команды).
- Проверка доверенности отправителя по единому предикату (-> rules-control).
- Классификация письма: ответ на bridged-письмо / команда вкл-выкл / игнорировать.
- Сопоставление ответа с записью связки по идентификатору исходного письма (In-Reply-To) → диалог.
- Отметка письма как потреблённого, чтобы не обработать его повторно.

## Не делает
- Не публикует в Telegram (это -> cmp-tg-gateway/cap-post-as-user) — отдаёт целевой диалог и текст.
- Не меняет состояние моста напрямую (это -> cmp-state-store) — отдаёт распознанную команду.
- Не отправляет подтверждения (это -> cmp-email-out/cap-send-notice).

## Возможности (capabilities)

### cap-poll-b-imap
- **Что делает:** возвращает новые (не потреблённые) письма на ящике моста B.
- **Вход (логически):** креды/адрес ящика B, множество уже потреблённых идентификаторов.
- **Выход (логически):** список новых писем (отправитель, адрес получателя, In-Reply-To, тема, тело).
- **Используется функциями АС/ФП:** -> fn-email-reply-to-tg, -> fn-bridge-control-by-email

### cap-authenticate-sender
- **Что делает:** применяет единый предикат доверия: отправитель = U И письмо пришло на B И является
  ответом на ранее доставленное (in-reply-to); для команды дополнительно — секрет-токен, если задан.
- **Вход (логически):** письмо; адрес U; наличие записи связки; сконфигурированный токен (опц.).
- **Выход (логически):** «доверено» / «отклонено» (с фиксацией факта отклонения в логе).
- **Используется функциями АС/ФП:** -> fn-email-reply-to-tg, -> fn-bridge-control-by-email

### cap-classify-message
- **Что делает:** относит доверенное письмо к типу: ответ на bridged-письмо (публикация в диалог) либо
  команда управления (вкл/выкл); нераспознанное — не исполняется.
- **Вход (логически):** доверенное письмо (тема/служебные признаки).
- **Выход (логически):** тип: reply / command(on|off) / ignore.
- **Используется функциями АС/ФП:** -> fn-email-reply-to-tg, -> fn-bridge-control-by-email

### cap-resolve-ledger
- **Что делает:** по идентификатору письма-родителя (In-Reply-To) находит запись связки и целевой диалог.
- **Вход (логически):** In-Reply-To ответа.
- **Выход (логически):** идентификатор целевого диалога либо «связки нет» → игнор.
- **Используется функциями АС/ФП:** -> fn-email-reply-to-tg

### cap-mark-consumed
- **Что делает:** фиксирует письмо потреблённым (после успешной публикации/исполнения) — повторный опрос
  или перезапуск не обрабатывает его снова (единожды).
- **Вход (логически):** идентификатор письма.
- **Выход (логически):** обновлённое множество потреблённых идентификаторов (ограниченное).
- **Используется функциями АС/ФП:** -> fn-email-reply-to-tg, -> fn-bridge-control-by-email

## Зависимости
- Ящик моста (B) как IMAP-источник (внешняя система) — -> api-mailbox-imap-smtp.
- -> cmp-state-store — запись связки (для in-reply-to → диалог), множество потреблённых, адрес U/токен.

## Связи
- Реализует функции АС/ФП: -> fn-email-reply-to-tg, -> fn-bridge-control-by-email
- Сценарии: -> scn-outbound-reply, -> scn-control-command
- Алгоритмы: -> alg-dedup-idempotency
- API: -> api-mailbox-imap-smtp
- Данные: -> data-bridge-store

## Evolve-дельта (13.07, живой тест)
**Устойчивая срезка цитаты (TKT-3/10).** `html_to_text` сохраняет переводы строк по блочным тегам
и целиком отбрасывает контейнер цитаты (`<blockquote>`, class~gmail_quote/moz-cite/cite) — в top-posting
всё после него принадлежит старой ветке. `strip_quoted_tail` затем режет остаточную атрибуцию по
мультилокальным маркерам (`_QUOTE_TAIL`: `>`, `… wrote:`, `… писал(а):`, баннеры пересылки, разделители).
Причина эволюции: старый вариант ловил только `On … wrote:` и схлопывал HTML в одну строку → русский
HTML-хвост просачивался в TG.

# Сценарий: Первичная авторизация Telegram (интерактивный setup → сессия)

## Цель сценария
Реализует однократную интерактивную авторизацию и порождение файла сессии для дальнейшей headless-работы
(гибридная авторизация). -> fn-first-run-setup, -> adr-002-telethon-hybrid-auth

## Триггер
Запуск интерактивной команды инициализации (первый запуск или отсутствие действительной сессии).

## Предусловия
Задана конфигурация доступа Telegram и путь к сессии; у пользователя есть телефон для приёма кода.

## Последовательность шагов
1. Прочитать конфигурацию доступа Telegram и путь к сессии. -> cmp-state-store/cap-read-config
2. Запросить телефон, код подтверждения и, при 2FA, пароль. -> cmp-tg-gateway/cap-surface-session-errors · -> api-telegram-userclient
3. Создать файл сессии с правами только для владельца (600) вне репозитория. -> cmp-state-store/cap-manage-session-health
4. Отметить сессию действительной И сбросить notified=false (чтобы будущий отзыв сессии снова уведомил U);
   фоновый сервис далее стартует неинтерактивно. -> cmp-state-store/cap-manage-session-health

## Ветвления
- Действительная сессия уже существует (шаг 1) → интерактивный шаг не требуется, сервис стартует headless;
  иначе → выполнить шаги 2–4.
- 2FA включена → запросить пароль; 2FA выключена → шаг ввода пароля пропускается.

## Ошибки и таймауты
- Неверный код/пароль (шаг 2) → авторизация не завершается, файл сессии не создаётся; явная ошибка и
  предложение повторить (интерактивно).
- Сессия отозвана/недействительна во время штатной работы → -> scn-session-invalid-alert (опрос остановлен,
  письмо-уведомление, блокирующее состояние до повторного запуска этого сценария).

## Постусловия
Существует файл сессии (права 600, вне репозитория), пригодный для неинтерактивной работы; состояние
сессии — «действительна», notified=false (готовность повторно уведомить при будущем отзыве).

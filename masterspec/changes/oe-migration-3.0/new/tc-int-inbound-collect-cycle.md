## Fidelity эксплуатационной проверки
- Источник OE-маппинга: -> scn-inbound-collect-cycle § «Реализация контракта живой эксплуатации»
- Профиль / fixture / внешняя среда: -> cmp-tg-gateway отдаёт заранее заданные сообщения по диалогам (заглушка -> api-telegram-userclient); -> cmp-email-out подключён к SMTP-заглушке ящика B (-> api-mailbox-imap-smtp), фиксирующей отправленные письма
- Fidelity: production-like
- Что реально проверяется: гейт адресованности (личка/белый список/упоминание), батчинг за такт, представление медиа и деградация по размеру, дедуп курсора, отказы выборки/скачивания/отправки
- Residual risk: реальный объём диалогов (эмпирика до 806) и реальная нагрузка провайдера почты проверяются отдельно нагрузочным прогоном по -> lp-dm-batch-to-email / -> lp-channel-update-to-email

## Предварительные действия
- Мост включён; сессия действительна; лимит размера письма задан так, что один набор вложений его превышает; курсоры диалогов на исходной позиции.

## Шаги выполнения
1. **Действие:** Запустите такт сбора при пакете сообщений D1 {m1, m2(изображение), m3(эхо), m4(вложение > порога)}, C1 (белый список) {c11}, C2 (вне списка, без упоминания) {c21}, C3 (вне списка, с упоминанием) {c31}.
   **Тестовые данные:** dialogs=D1,C1,C2,C3; fetch-mode=success
   **Ожидаемый результат:**
   - На ящик B уходит по одному письму на D1, C1, C3; письма для C2 нет (гейт отсеял).

2. **Действие:** Проверьте состав и порядок письма диалога D1.
   **Тестовые данные:** dialog=D1
   **Ожидаемый результат:**
   - Письмо содержит m1, m2, m4 в хронологическом порядке; m3 (эхо) отсутствует; изображение m2 отображено инлайн; вложение m4 (> порога) представлено текстовым указанием.

3. **Действие:** Задайте набор вложений диалога D1, при котором итоговое письмо превышает лимит размера провайдера, и повторите формирование письма.
   **Тестовые данные:** dialog=D1; attachment-set=oversize
   **Ожидаемый результат:**
   - Крупные вложения заменены текстовым указанием; при остаточном превышении письмо раздроблено на несколько писем с сохранённым порядком сообщений; ни одно сообщение не потеряно.

4. **Действие:** Переведите выборку Telegram для D1 в режим unavailable (FloodWait/transient) и запустите такт.
   **Тестовые данные:** dialog=D1; fetch-mode=unavailable
   **Ожидаемый результат:**
   - Такт для D1 пропущен, курсор D1 НЕ продвинут; письма для D1 на этом такте нет.

5. **Действие:** Верните выборку в режим success и запустите следующий такт для D1.
   **Тестовые данные:** dialog=D1; fetch-mode=success
   **Ожидаемый результат:**
   - Сообщения D1 пересобраны с сохранённого курсора без потери и без дубля уже доставленных.

6. **Действие:** Переведите скачивание вложения m4 диалога D1 в режим media-unavailable и сформируйте письмо.
   **Тестовые данные:** dialog=D1; media=m4; download-mode=media-unavailable
   **Ожидаемый результат:**
   - Вложение m4 представлено текстовым указанием «вложение недоступно»; письмо всё равно доставлено, текст сообщения не блокируется.

7. **Действие:** Переведите отправку письма C1 в режим сбоя (SMTP-транзиент) и запустите такт.
   **Тестовые данные:** dialog=C1; send-mode=unavailable
   **Ожидаемый результат:**
   - Курсор C1 НЕ продвинут, запись связки не создаётся; письмо не отправлено на этом такте.

8. **Действие:** Верните отправку в режим success и запустите следующий такт для C1.
   **Тестовые данные:** dialog=C1; send-mode=success
   **Ожидаемый результат:**
   - Сообщения C1 доставлены без дубля для уже доставленных и без потери недоставленных на шаге 7.

9. **Действие:** Проверьте диагностический журнал по идентификатору операции и окну времени для отказов шагов 4, 6 и 7.
   **Тестовые данные:** correlation-id=int-inbound-001; time-window=5 минут; expected-event=fetch-retry-scheduled|media-unavailable|delivery-retry-scheduled
   **Ожидаемый результат:**
   - Найдена ровно одна запись `fetch-retry-scheduled` (D1, шаг 4), одна `media-unavailable` (D1/m4, шаг 6) и одна `delivery-retry-scheduled` (C1, шаг 7), каждая со статусом `pending`/`degraded` и специфицированной причиной.

## Проверяемые условия
- Число писем за такт 1 = 3 (D1, C1, C3); C2 не порождает письма (гейт).
- В письме D1 порядок сообщений хронологический; эхо исключено; изображение инлайн; крупное вложение — указание; итоговое письмо ≤ лимита размера (деградация/дробление сработали).
- Повторный такт/перезапуск не создаёт дублей доставленных сообщений (0 дублей); при сбоях выборки/отправки курсор не продвинут до успеха.

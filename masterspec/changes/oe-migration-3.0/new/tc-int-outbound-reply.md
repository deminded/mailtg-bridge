- Грани живой эксплуатации: -> fn-email-reply-to-tg/OE-EVIDENCE, -> fn-email-reply-to-tg/OE-RESILIENCE, -> fn-email-reply-to-tg/OE-SOURCES

- Профиль / fixture / внешняя среда: -> cmp-email-in подключён к IMAP-заглушке ящика B (-> api-mailbox-imap-smtp); -> cmp-tg-gateway с публикацией-заглушкой (-> api-telegram-userclient), умеющей смоделировать сбой публикации и session-invalid

- Residual risk: реальная публикация в Telegram проверяется отдельно live-e2e (-> tc-acc-email-reply)

## Предварительные действия
- Мост включён; сессия действительна. В журнале связки есть запись для bridged-письма диалога D (Message-ID=MID). IMAP-заглушка B и публикация-заглушка Telegram — в режиме success.

## Шаги выполнения
1. **Действие:** Подайте ответ R1 с адреса U на bridged-письмо (in_reply_to=MID).
   **Тестовые данные:** R1={from=U, in_reply_to=MID, text="ответ"}
   **Ожидаемый результат:**
   - В диалоге D появляется одно сообщение от лица пользователя с текстом «ответ».

2. **Действие:** Подайте ответ R2 с чужого адреса на то же bridged-письмо.
   **Тестовые данные:** R2={from=чужой@example.com, in_reply_to=MID}
   **Ожидаемый результат:**
   - Публикации в Telegram нет.

3. **Действие:** Подайте ответ R3 с адреса U на письмо без записи связки (не-bridged).
   **Тестовые данные:** R3={from=U, in_reply_to=неизвестный-message-id}
   **Ожидаемый результат:**
   - Публикации в Telegram нет.

4. **Действие:** Повторно подайте R1 в опрос (тот же ответ, повторный такт/перезапуск).
   **Тестовые данные:** R1'=R1 (то же письмо, повторный опрос)
   **Ожидаемый результат:**
   - Повторной публикации в Telegram нет (0 дублей); письмо остаётся потреблённым единожды.

5. **Действие:** Переведите IMAP-заглушку ящика B в режим unavailable и запустите такт опроса.
   **Тестовые данные:** mailbox-mode=poll:unavailable
   **Ожидаемый результат:**
   - Такт пропущен; письма на B не теряются (остаются необработанными); публикаций нет.

6. **Действие:** Переведите IMAP-заглушку ящика B в режим auth-failed (неверные креды) и запустите такт опроса.
   **Тестовые данные:** mailbox-mode=poll:auth-failed
   **Ожидаемый результат:**
   - Явная ошибка такта; такт не выполняется; секреты в ошибке отсутствуют.

7. **Действие:** Верните ящик B в режим success, подайте новый ответ R4 с адреса U на bridged-письмо и переведите публикацию в Telegram в режим сбоя (FloodWait/transient).
   **Тестовые данные:** R4={from=U, in_reply_to=MID, text="второй ответ"}; post-mode=unavailable
   **Ожидаемый результат:**
   - Ответ R4 НЕ помечается потреблённым; повторная попытка публикации запланирована на следующий такт по backoff.

8. **Действие:** Переведите публикацию для R4 в режим session-invalid и запустите такт.
   **Тестовые данные:** post-mode=session-invalid
   **Ожидаемый результат:**
   - Отказ трактуется консервативно как session-invalid: сервис переходит к -> scn-session-invalid-alert (публикация не выполняется, дальнейший опрос останавливается).

9. **Действие:** Проверьте диагностический журнал по идентификатору операции и окну времени для отказов R2, R3, шагов 5, 6 и 7.
   **Тестовые данные:** correlation-id=int-outbound-001; time-window=5 минут; expected-event=reply-rejected|poll-retry-scheduled|post-retry-scheduled
   **Ожидаемый результат:**
   - Найдены записи `reply-rejected` для R2 (чужой адрес) и R3 (нет записи связки) без содержания тела; запись `poll-retry-scheduled` для шага 5; запись `post-retry-scheduled` для R4 (шаг 7), каждая со статусом `pending`/`rejected` и специфицированной причиной.

## Проверяемые условия
- Публикаций в Telegram = 1 после шага 1 (только R1); R2, R3, R1' (шаг 4) не порождают публикаций.
- Маршрутизация — в диалог D по записи связки (по диалогу, не по сообщению).
- Недоверенное письмо (R2) и не-bridged письмо (R3) зафиксированы в логе без содержания тела (-> nfr-privacy).
- Отметка «потреблено» ставится только после успешной публикации (R4 не потреблён на шаге 7).

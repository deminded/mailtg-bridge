# Интеграционный тест: управление мостом письмом (доверие + токен + идемпотентность)

## Проверяет
- Сценарий: -> scn-control-command
- Функция: -> fn-bridge-control-by-email
- Путь: основной + ветвления «недоверенный отправитель», «неверный/отсутствующий токен», «идемпотентная команда»; ошибки «недоступность ящика при опросе», «сбой отправки подтверждения»
- Грани живой эксплуатации: -> fn-bridge-control-by-email/OE-EVIDENCE, -> fn-bridge-control-by-email/OE-RESILIENCE, -> fn-bridge-control-by-email/OE-SOURCES, -> fn-bridge-control-by-email/OE-SECURITY
- Строки каталога: -> tc-flt-control-command/FLT-000, -> tc-flt-control-command/FLT-001, -> tc-flt-control-command/FLT-002

## Fidelity эксплуатационной проверки
- Источник OE-маппинга: -> scn-control-command § «Реализация контракта живой эксплуатации»
- Профиль / fixture / внешняя среда: IMAP/SMTP-заглушка ящика B с управляемыми режимами success/unavailable/tech-error; -> cmp-state-store хранит состояние моста
- Fidelity: production-like
- Что реально проверяется: предикат доверия+токен, идемпотентность команды, применение состояния, отправка подтверждения, отказ ящика на опросе и на отправке подтверждения
- Residual risk: конечная доставка письма реальным почтовым провайдером проверяется отдельно live-e2e (-> tc-acc-deploy-and-security)

## Предварительные действия
- Сконфигурирован адрес U и секрет-токен темы; мост в состоянии «включён»; IMAP/SMTP-заглушка ящика B в режиме success.

## Шаги выполнения
1. **Действие:** Подайте письмо-команду K1 «выключить» с адреса U на ящик B с верным токеном.
   **Тестовые данные:** K1={from=U, subject="MAILTG OFF <token>", in_reply_to=доставленное письмо}
   **Ожидаемый результат:**
   - Мост переходит в состояние «выключено»; на U отправлено письмо-подтверждение.

2. **Действие:** Подайте письмо-команду K2 «включить» с чужого адреса (не U) с верным токеном.
   **Тестовые данные:** K2={from=чужой@example.com, subject="MAILTG ON <token>"}
   **Ожидаемый результат:**
   - Состояние моста не меняется; подтверждение не отправлено.

3. **Действие:** Подайте письмо-команду K3 «включить» с адреса U с отсутствующим/неверным токеном.
   **Тестовые данные:** K3={from=U, subject="MAILTG ON"}
   **Ожидаемый результат:**
   - Состояние моста не меняется; подтверждение не отправлено.

4. **Действие:** Подайте письмо-команду K4 «выключить» повторно с адреса U с верным токеном на уже выключенном мосте.
   **Тестовые данные:** K4={from=U, subject="MAILTG OFF <token>"}
   **Ожидаемый результат:**
   - Состояние моста остаётся «выключено» (не меняется повторно); подтверждение отправлено идемпотентно.

5. **Действие:** Перезапустите сервис и прочитайте состояние моста.
   **Тестовые данные:** N/A — чтение состояния после рестарта без нового ввода
   **Ожидаемый результат:**
   - Состояние «выключено» сохранено после перезапуска.

6. **Действие:** Переведите IMAP-заглушку ящика B в режим unavailable и подайте команду K5 «включить» с адреса U с верным токеном.
   **Тестовые данные:** K5={from=U, subject="MAILTG ON <token>"}; mailbox-mode=poll:unavailable
   **Ожидаемый результат:**
   - Такт пропущен, письмо-команда K5 остаётся необработанным на B; состояние моста не меняется.

7. **Действие:** Верните опрос в режим success, переведите отправку подтверждения в режим tech-error и повторите такт для K5.
   **Тестовые данные:** mailbox-mode=poll:success,send:tech-error
   **Ожидаемый результат:**
   - Мост переходит в состояние «включено» (применение состояния успешно), но отправка подтверждения не удалась технически; повтор отправки подтверждения запланирован на следующий такт по backoff.

8. **Действие:** Проверьте диагностический журнал по идентификатору операции и окну времени для K2, K3 и такта unavailable (K5).
   **Тестовые данные:** correlation-id=int-control-001; time-window=5 минут; expected-event=command-rejected|mailbox-unavailable
   **Ожидаемый результат:**
   - Найдены записи `command-rejected` (K2 — чужой адрес, K3 — неверный токен) со статусом `rejected` и специфицированной причиной; запись `mailbox-unavailable` для такта K5 со статусом `pending`.

## Проверяемые условия
- Итоговое состояние моста = «включён» (после шага 7).
- Подтверждения отправлены только на доверенные исполненные команды (K1; K4 — идемпотентное подтверждение).
- Недоверенная (K2) и непрошедшая токен (K3) команды не меняют состояние и залогированы.
- Состояние переживает перезапуск (шаг 5).
- Недоступность ящика на опросе (K5/unavailable) не меняет состояние и не создаёт дубля; сбой отправки подтверждения не теряет применённое состояние.

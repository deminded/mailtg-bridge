# Функция: Доставка обновлений каналов и групп по почте (белый список или упоминание)

## Цель
Не терять важные обновления каналов и групп, отфильтровав шум: письмом приходит только то, что в
белом списке или прямо упоминает пользователя.

## Инициатор
Расписание (такт сбора входящих).
- **Внешний инициатор/канал:** Каналы и группы Telegram, отслеживаемые через личный аккаунт
  пользователя (белый список / список упоминаний / все диалоги при `DISCOVER_ALL_DIALOGS`),
  опрашиваемые по расписанию; адрес пользователя U — приёмник письма (через ящик моста B, SMTP).

## Триггер
Наступление такта сбора при наличии новых обновлений каналов/групп после курсора диалога.

## Предусловия
Мост включён; сессия действительна; в конфигурации заданы белый список и политика упоминаний.

## Основной поток
1. Система опрашивает Telegram и получает новые обновления отслеживаемых каналов/групп после курсора.
2. Система применяет гейт: пропускает обновление, если источник в белом списке ИЛИ есть прямое
   упоминание пользователя; иначе отсеивает.
3. Система группирует пропущенные обновления каждого диалога (канал/группа/топик) в батч за такт.
4. Система формирует одно письмо на диалог (HTML) с тегом источника и списком сообщений; для каждого
   — автор (имя и @username), локализованное время, текст с сохранённым форматированием, цитата-ответ
   и глубокая ссылка (deep-link) на оригинал в Telegram (публичные — `t.me/<user|channel>/<msg>`,
   приватные — `t.me/c/<chat>/<msg>`, топик — в свой топик); медиа — через -> fn-media-in-email.
   Правила представления и ссылки — -> rules-delivery.
5. Система отправляет письмо на адрес пользователя U (через ящик моста B, SMTP), фиксирует запись
   связки и продвигает курсор диалога.

## Альтернативные потоки
- Обновление из невайтлист-источника без упоминания → не доставляется (поведение по умолчанию).
- Политика упоминаний в режиме форсирования по списку → упоминания учитываются только для каналов
  из списка согласно конфигурации (все/выбранные/по списку).
- Несколько обновлений одного диалога за такт → одно письмо со списком.

## Исключительные потоки (ошибки, отказы)
- Источник недоступен для опроса → такт для него пропускается без потери курсора; наблюдаемый исход:
  сбор продолжается при восстановлении доступа.
- Отправка письма не удалась → курсор не продвигается, доставка откладывается без дубля.
- Мост выключен → сбор и доставка не выполняются.

## Результат
Письмо-батч на диалог для адресованных обновлений; запись связки; продвинутый курсор.

## Постусловия
Только адресованные обновления доставлены; неадресованные — отсеяны без письма; дублей нет.

## Контракт живой эксплуатации (OE)

### OE-LOAD — объём и ритм
- **Статус:** APPLICABLE
- **Контракт:** Популяция — сконфигурированный набор отслеживаемых источников (белый список ∪
  список упоминаний), либо все диалоги аккаунта при `DISCOVER_ALL_DIALOGS=true` и
  `MENTION_POLICY=all`; активное подмножество — источники с новыми сообщениями после курсора; частота
  такта — `COLLECT_INTERVAL_SECONDS` (дефолт 60 с); выборка за такт ограничена `TG_FETCH_LIMIT`
  (дефолт 100/диалог); backlog после простоя пересобирается без потери. Тот же инвариант
  масштабирования, что и для личных сообщений: round-trips за такт ≈ O(активные), не O(все) —
  использует общий цикл опроса (-> fn-dm-batch-to-email/OE-LOAD).
- **Основание:** код: `config.py` (`DISCOVER_ALL_DIALOGS`, `COLLECT_INTERVAL_SECONDS`,
  `TG_FETCH_LIMIT`), `telegram.py` (`list_tracked_dialogs` — два режима отбора источников);
  -> nfr-operability.
- **Проверка:** AC-07

### OE-INPUT — реальные формы входа
- **Статус:** APPLICABLE
- **Контракт:** То же множество форм сообщений, что и для личных (текст/entities/цитата/автор), плюс
  различение источника: публичный канал/группа (есть `username`), приватная супергруппа/канал (только
  `peer_id`, доступно только участнику), топик форума (`topic_id`) — определяет форму глубокой
  ссылки.
- **Основание:** код: `telegram.py` (`_dialog` — ветки `username`/`peer_id`/`topic_id`);
  -> rules-delivery.
- **Проверка:** AC-05, AC-06

### OE-EVIDENCE — наблюдаемый исход и след
- **Статус:** APPLICABLE
- **Контракт:** Успех — письмо с тегом источника/диалога приходит в ящик U в пределах такта; глубокая
  ссылка — проверяемое пользователем доказательство (клик открывает то самое сообщение/топик).
  Неполучение в текущем такте не является потерей: курсор не продвинут при сбое, сообщение приходит
  на следующем такте без дубля.
- **Основание:** код: `orchestrator.py` (`commit_delivery`); -> rules-delivery.
- **Проверка:** AC-01, AC-05

### OE-SOURCES — классы источников и политика
- **Статус:** APPLICABLE
- **Контракт:** Доставка ТОЛЬКО при попадании источника в белый список ИЛИ прямом упоминании
  пользователя (режимы политики упоминаний: все/выбранные/форсирование по списку); по умолчанию —
  отказ (не воспроизводить шум). Гейт по политике ботов (`bot_policy`) применяется ДО проверки
  вайтлиста/упоминания — сообщение от незалистенного бота отсеивается даже из вайтлист-канала. Общая
  формула — -> rules-gating.
- **Основание:** -> rules-gating (включая эволюцию TKT-13); код: `algorithms.py` (`is_addressed`).
- **Проверка:** AC-01, AC-02, AC-03, AC-08

### OE-SECURITY — конфиденциальность и целостность
- **Статус:** APPLICABLE
- **Контракт:** Как для личных сообщений: источник аутентифицирован сессией, назначение письма
  фиксировано конфигурацией, контентная фильтрация (по смыслу) вне скоупа продукта
  (-> as-mailtg-bridge/Что не умеет), тела сообщений и секреты не попадают в логи. Общая политика —
  -> rules-security, -> nfr-security.
- **Основание:** код: `logging.py`; -> rules-security.
- **Проверка:** AC-09

### OE-RESILIENCE — повтор, отказ и восстановление
- **Статус:** APPLICABLE
- **Контракт:** Идемпотентность курсор+журнал, как для личных сообщений; недоступность источника
  (FloodWait/сетевая ошибка) → такт для этого источника пропускается без потери курсора, со скоуп-
  backoff (`BACKOFF_MIN_SECONDS`=30…`BACKOFF_MAX_SECONDS`=3600, джиттер ±10%); сбой одного источника
  не блокирует остальные; та же файловая блокировка единственного экземпляра, что и для остальных
  функций. Общая формула — -> rules-integrity.
- **Основание:** код: `orchestrator.py` (скоуп `tg:<dialog_id>`, `record_failure`), `locking.py`.
- **Проверка:** AC-10

### OE-DELIVERY — сквозной путь результата
- **Статус:** APPLICABLE
- **Контракт:** Доставлено = SMTP-приём почтовым сервером B (transport ack), тот же предел
  подтверждения, что и для личных сообщений (bounce/read-receipt не отслеживаются, декларированный
  остаточный риск). Сбой отправки → курсор не продвинут → передоставка на следующем такте без
  потери. При превышении лимита размера — деградация (-> fn-media-in-email, -> rules-delivery).
- **Основание:** код: `mail_out.py` (`SmtpMailer.send`); -> rules-delivery.
- **Проверка:** AC-11

### OE-CONTROL — живой управляющий ввод
- **Статус:** N/A — функция только формирует и отправляет письмо по расписанию; управляющую
  человеческую грамматику она не парсит.
- **Контракт:** N/A — распознавание команд принадлежит -> fn-bridge-control-by-email.
- **Основание:** код: `orchestrator.py` (`run_inbound_cycle` не вызывает `parse_command`).
- **Проверка:** N/A

## Критерии приёмки
- AC-01: дано сообщение из вайтлист-источника, когда наступает такт, тогда оно доставляется письмом
  с тегом источника.
- AC-02: дано сообщение с прямым упоминанием пользователя из невайтлист-источника, когда наступает
  такт, тогда оно доставляется письмом.
- AC-03: дано сообщение из невайтлист-источника без упоминания, когда наступает такт, тогда оно не
  доставляется (по умолчанию).
- AC-04: дано изменение политики доставки, когда меняется конфигурация, тогда новое поведение
  вступает в силу без изменения кода.
- AC-05: дано доставленное обновление канала/группы/топика, когда пользователь кликает его глубокую
  ссылку, тогда в Telegram открывается именно это сообщение (публичные — `t.me/<user|channel>/<msg>`,
  приватные — `t.me/c/<chat>/<msg>`, топик — в свой топик).
- AC-06: дано сообщение с форматированием, автором и цитатой-ответом, когда формируется письмо, тогда
  получатель опознаёт автора, локализованное время и структуру как в клиенте без потери форматирования.
- AC-07: дано настроенный набор отслеживаемых каналов/групп и простой перед тактом, когда наступает
  такт после простоя, тогда фетчатся только источники с `top_id` выше сохранённого курсора
  (round-trips ≈ O(активные)), backlog пересобирается без потери на последующих тактах.
- AC-08: дано сообщение от бота в отслеживаемом (вайтлист) канале при `bot_policy=selected` и бот не
  входит в `bot_list`, когда наступает такт, тогда сообщение не доставляется, несмотря на попадание
  канала в белый список.
- AC-09: дано такт обработал тела сообщений канала/группы, когда логи такта инспектируются, тогда
  логи содержат диалог/тег/счётчики обработки, но не текст сообщений и не секреты (0 вхождений).
- AC-10: дано источник временно недоступен (FloodWait/сетевая ошибка), когда такт выполняется, тогда
  такт для этого источника пропускается без потери курсора и с соблюдением backoff; сбор
  продолжается при восстановлении доступа.
- AC-11: дано отправка письма-батча канала/группы не удалась, когда такт завершается, тогда курсор
  диалога не продвинут и на следующем такте сообщения доставлены без потери и без дубля.

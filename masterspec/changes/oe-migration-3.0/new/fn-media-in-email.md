# Функция: Представление медиа и вложений Telegram в письме

## Цель
Показать содержимое медиа из Telegram в письме удобно и без раздувания ящика: изображения — сразу
видны в теле, прочие вложения — по размеру.

## Инициатор
Система (в ходе формирования письма-батча).
- **Внешний инициатор/канал:** Медиасерверы Telegram — источник вложений, скачиваемых через личный
  аккаунт пользователя в ходе формирования письма-батча; встроенное/приложенное представление уходит
  в письме на адрес пользователя U (через ящик моста B, SMTP).

## Триггер
Наличие у сообщения батча вложений (изображений и/или прочих файлов).

## Предусловия
Формируется письмо-батч диалога; заданы порог размера вложений и лимит размера письма (провайдера)
в конфигурации.

## Основной поток
1. Для каждого изображения сообщения система встраивает его в тело письма (инлайн, в HTML-теле).
2. Для каждого не-изображения размером не выше порога система прикладывает его файлом.
3. Для каждого не-изображения размером выше порога система добавляет текстовое указание
   (имя / размер / тип) без тела вложения.
4. Система собирает тело письма как HTML со встроенными изображениями и текстовым запасным вариантом.

## Альтернативные потоки
- Сообщение без вложений → тело содержит только текст (с запасным текстовым вариантом).
- Несколько изображений в сообщении → все встраиваются инлайн, порядок сохраняется.
- (Задел версии 2) крупное вложение → ссылка на веб-диск вместо текстового указания.

## Исключительные потоки (ошибки, отказы)
- Скачивание вложения из Telegram не удалось → в письмо помещается указание о недоступном вложении;
  наблюдаемый исход: письмо всё равно доставлено, текст не блокируется, вложение помечено
  недоступным.
- Изображение не удалось встроить → оно понижается до текстового указания (имя/тип), доставка тела
  не срывается.
- Письмо превысило бы лимит размера почтового провайдера → деградация: крупные вложения переводятся
  в текстовое указание (или опускаются), при остаточном превышении батч дробится на несколько писем;
  доставка не срывается и ничего не теряется молча (-> rules-delivery).

## Результат
Письмо, где изображения видны в теле, малые вложения приложены файлами, крупные — обозначены
текстовым указанием.

## Постусловия
Тело письма — валидный HTML с текстовым запасным вариантом; размер письма не превышает лимит
провайдера (при необходимости достигнут деградацией/дроблением).

## Контракт живой эксплуатации (OE)

### OE-LOAD — объём и ритм
- **Статус:** APPLICABLE
- **Контракт:** Популяция — вложения сообщений батча текущего такта (0..N на сообщение); порог
  файл/указание — `ATTACHMENT_THRESHOLD_BYTES` (дефолт 10 МБ), лимит письма —
  `EMAIL_SIZE_LIMIT_BYTES` (дефолт ~24 МБ), оба конфигурируемы; рост пропорционален числу медиа-
  сообщений батча диалога, не является отдельным от -> fn-dm-batch-to-email/OE-LOAD /
  -> fn-channel-update-to-email/OE-LOAD тактом.
- **Основание:** код: `config.py` (`ATTACHMENT_THRESHOLD_BYTES`, `EMAIL_SIZE_LIMIT_BYTES`).
- **Проверка:** AC-05

### OE-INPUT — реальные формы входа
- **Статус:** APPLICABLE
- **Контракт:** Реальные формы вложений Telegram: изображения (`content_type` `image/*`), прочие
  файлы произвольного MIME-типа; размер (`MediaRef.size`) может быть неизвестен до скачивания.
  Скачивание — во временный каталог, файл удаляется после отправки письма независимо от исхода.
- **Основание:** код: `telegram.py` (`download_media`, `MediaRef`), `orchestrator.py` (`finally:
  item.path.unlink(missing_ok=True)`).
- **Проверка:** AC-01, AC-02, AC-03

### OE-EVIDENCE — наблюдаемый исход и след
- **Статус:** APPLICABLE
- **Контракт:** Успех — вложение видно в письме (инлайн / файлом / текстовым указанием) в
  соответствии с его классом и размером. Отказ скачивания — явное текстовое указание «вложение
  недоступно» в письме (не пропуск молча), само письмо при этом доставлено.
- **Основание:** код: `mail_out.py` (`_render` — маркер `[omitted media: ...]`).
- **Проверка:** AC-06

### OE-SOURCES — классы источников и политика
- **Статус:** N/A — функция не гейтит по классам отправителей сообщений; адресованность уже решена
  выше по потоку (-> rules-gating, применяется в -> fn-dm-batch-to-email / -> fn-channel-update-to-email
  до вызова этой функции). Здесь обрабатываются медиа уже адресованных сообщений без собственной
  политики источников.
- **Контракт:** N/A
- **Основание:** код: `orchestrator.py` (медиа скачивается только для уже отобранных `selected`
  сообщений, гейт `is_addressed` применён раньше).
- **Проверка:** N/A

### OE-SECURITY — конфиденциальность и целостность
- **Статус:** APPLICABLE
- **Контракт:** Скачанные файлы временные — не остаются в постоянном хранилище вне письма; содержимое
  вложений не логируется. Общая политика — -> rules-security, -> nfr-privacy.
- **Основание:** код: `orchestrator.py` (удаление временного файла в `finally`), `config.py`
  (`TEMP_DIR`, права каталога 0o700).
- **Проверка:** AC-07

### OE-RESILIENCE — повтор, отказ и восстановление
- **Статус:** APPLICABLE
- **Контракт:** Неуспешное встраивание изображения деградирует до текстового указания без срыва
  письма; превышение лимита размера письма деградирует по порядку (крупные вложения → текстовое
  указание; при остаточном превышении — дробление батча на несколько писем с сохранением порядка
  сообщений), не теряя сообщений молча.
- **Основание:** код: `mail_out.py` (`compose_batch` — цикл деления `groups`/`oversized`,
  `MailSizeRejected` только как крайний случай).
- **Проверка:** AC-05

### OE-DELIVERY — сквозной путь результата
- **Статус:** APPLICABLE
- **Контракт:** Доставлено — как часть письма-батча диалога (-> fn-dm-batch-to-email/OE-DELIVERY,
  -> fn-channel-update-to-email/OE-DELIVERY); эта функция гарантирует, что раздутие или недоступность
  медиа не создают молчаливой потери письма целиком — применяется деградация вместо срыва. Общая
  политика — -> rules-delivery.
- **Основание:** код: `mail_out.py` (`compose_batch` — `MailSizeRejected` только после исчерпания
  деградации); -> rules-delivery.
- **Проверка:** AC-05

### OE-CONTROL — живой управляющий ввод
- **Статус:** N/A — функция только представляет данные (медиа) в письме; управляющего человеческого
  ввода не принимает.
- **Контракт:** N/A
- **Основание:** код: модуль не содержит разбора команд, только рендеринг вложений.
- **Проверка:** N/A

## Критерии приёмки
- AC-01: дано сообщение с изображением, когда формируется письмо, тогда изображение отображается
  встроенным в теле письма (не приложением).
- AC-02: дано не-изображение размером ≤ порога, когда формируется письмо, тогда оно приходит
  приложенным файлом.
- AC-03: дано не-изображение размером > порога, когда формируется письмо, тогда оно приходит
  текстовым указанием (имя/размер/тип) без тела.
- AC-04: дано изменение порога в конфигурации, когда формируется письмо, тогда граница
  файл/указание меняется без изменения кода.
- AC-05: дано набор вложений, при котором письмо превысило бы лимит размера провайдера, когда
  формируется письмо, тогда крупные вложения приходят текстовым указанием (или батч дробится),
  письмо доставляется и ничего не теряется молча.
- AC-06: дано скачивание вложения из Telegram не удалось (медиа недоступно/временная ошибка), когда
  формируется письмо, тогда вложение заменяется текстовым указанием о недоступности и письмо всё
  равно доставлено (текст сообщения не блокируется).
- AC-07: дано вложение скачано во временный каталог для формирования письма, когда письмо отправлено
  (успешно или нет), тогда временный файл удалён и не остаётся в постоянном хранилище.

# Входные бизнес-требования — tg-mail-bridge (generic, open-source)

Контекст. Пользователь хочет оставаться на связи со своим Telegram через ПОЧТУ: когда он вдали
от TG или включил белые списки, важные сообщения (личка, упоминания в каналах/группах) должны
приходить письмом, а ответ письмом — уходить обратно в Telegram ОТ ЕГО ЛИЦА. Бот не годится:
он не видит личку и не отвечает «от себя», поэтому TG-сторона — юзерклиент MTProto (Telethon).
Продукт — открытый провайдер-нейтральный репозиторий. Обкатанное ядро (переиспользуем):
мой channel-reader (Telethon-поллинг каналов), bridge_poller (гейт/роутинг), de-agent-commons
(IMAP/SMTP), MessageLedger (дедуп/связка).

---
## TKT-TMB-1 — Личка Telegram → почта
**AS IS.** При whitelist-режиме или вдали от TG пользователь не видит личные сообщения.
**TO BE.** Каждое новое сообщение в личке TG-аккаунта (через Telethon) доставляется письмом на настроенный ящик: отправитель, текст, время; помечено источником (напр. `dm:<sender>`).
**Критерий приёмки** (наблюдаемый). Новое DM приходит письмом в пределах интервала поллинга с корректными отправителем/текстом; одно сообщение не дублируется даже после рестарта.
**Скоуп.** Личка. Много источников → один ящик, тег источника в письме. НЕ входит: медиа-рендер (см. TKT-4).
**Изменение БД / API / интеграции.** БД: нет (ledger + per-dialog last-id). API: Telethon (MTProto), SMTP. Интеграции: Telegram (user-session), SMTP-сервер.
**Релиз.** v0.1.

## TKT-TMB-2 — Каналы/группы → почта (вайтлист или упоминание)
**AS IS.** Важные апдейты в каналах/группах теряются среди шума или не доходят.
**TO BE.** Обновления из каналов/групп доставляются письмом ТОЛЬКО если источник в вайтлисте ИЛИ есть прямое упоминание пользователя. Настраивается: все/выбранные; форсирование упоминаний да/нет/выборочно по списку каналов.
**Критерий приёмки.** Сообщение из вайтлист-источника или с @упоминанием → письмо; из невайтлист-источника без упоминания → не доставляется (дефолт). Смена политики — только правкой конфига.
**Скоуп.** Каналы + группы. НЕ входит: контентная фильтрация.
**Изменение БД / API / интеграции.** БД: нет. API: Telethon, SMTP. Интеграции: Telegram.
**Релиз.** v0.1.

## TKT-TMB-3 — Ответ из почты → Telegram (от лица пользователя)
**AS IS.** Ответить на TG-сообщение можно только внутри Telegram.
**TO BE.** Ответ письмом на доставленное bridged-сообщение → постится в исходный TG-чат ОТ ЛИЦА пользователя (Telethon), в правильный диалог (по ledger).
**Критерий приёмки.** Reply письмом на bridged-письмо → сообщение появляется в нужном TG-чате от пользователя, в правильном диалоге; ответ на не-bridged письмо игнорируется.
**Скоуп.** Текстовый ответ. НЕ входит: вложения из письма в TG (v0.1).
**Изменение БД / API / интеграции.** БД: нет. API: IMAP (приём reply), Telethon (отправка). Интеграции: Telegram, IMAP-сервер.
**Релиз.** v0.1.

## TKT-TMB-4 — Вложения по размеру
**AS IS.** —
**TO BE.** TG-вложения в письме по размеру: маленькие (≤ порог из конфига) — приложенными файлами; крупные — указанием (имя/размер/тип) без тела; позже — ссылкой на веб-диск для скачивания.
**Критерий приёмки.** Файл ≤ порога приходит вложением; > порога — строкой-указанием; порог настраивается в конфиге.
**Скоуп.** TG→email вложения. НЕ входит (v0.1): выкладка на веб-диск (задел v0.2).
**Изменение БД / API / интеграции.** БД: нет. API: Telethon (download_media), SMTP (attach). Интеграции: (v0.2) веб-диск.
**Релиз.** v0.1 (порог/указание), v0.2 (веб-диск).

## TKT-TMB-8 — Управление мостом письмом (вкл/выкл) — добавлено после старта derive, для evolve
**AS IS.** —
**TO BE.** Пользователь управляет мостом командами по email с доверенного адреса: включить/выключить доставку (задел — пауза, статус, смена периодичности). Команда-письмо на служебный адрес/с служебной темой → мост меняет состояние и шлёт подтверждение письмом.
**Критерий приёмки.** Письмо-команда «выключить» с доверенного адреса → мост перестаёт доставлять и постить, отвечает подтверждением; «включить» → возобновляет; команда с НЕдоверенного адреса игнорируется (и логируется).
**Скоуп.** Вкл/выкл (v0.1). Аутентификация команды: отправитель из allowlist + опц. секрет-токен в теме. НЕ входит (v0.1): полноценный конфиг-по-почте.
**Изменение БД / API / интеграции.** БД: нет (файл состояния on/off). API: IMAP (приём команд). Интеграции: —.
**Релиз.** v0.1.

## РЕШЕНИЕ по гранулярности доставки (12.07, уточняет TKT-1/2/3/5)
**Батчинг, не письмо-на-сообщение.** Единица доставки = БАТЧ одного диалога (чат/топик или отправитель личка) за такт поллинга: одно письмо на диалог за цикл, внутри — список сообщений (от кого / когда / текст) в порядке. Иначе busy-чат = почтовый флуд.
**Следствие для ответа (закрывает Q5):** reply-to конкретного TG-сообщения НЕ нужен. Ответ письмом уходит в ДИАЛОГ целиком; `ledger` ключуется по диалогу (чат/топик), не по одному сообщению. Ответ на конкретное сообщение — задел v0.2 (маркер сообщения в теле письма + reply на маркер).
**Правки, которые это вносит:** TKT-1/TKT-2 «каждое сообщение → письмо» → «батч диалога за такт → письмо»; TKT-3 маршрутизация ответа = по диалогу; TKT-5 ledger на диалог.

## Технологические решения (adr-кандидаты, согласовано Евгением 12.07)
- **Язык: Python.** Переиспользуем обкатанное питоновское ядро (Telethon/agentcommons/ledger); Telethon зрелее rust-grammers на чувствительной user-сессии; нагрузка I/O-bound — скорость Rust не включается; опенсорс-доступность + гладкая передача Codex.
- **Авторизация Telethon: гибрид.** Интерактивная команда `setup`/`login` (телефон→код→2FA) рождает session-файл ОДИН раз (локально или `docker run -it … setup`); сервис-поллер потребляет session неинтерактивно (headless-деплой). Обкатано на arete-userbot.
- Сводка решённых развилок: поллинг (не демон); много TG-источников → один ящик (тег источника, роутинг ответа по ledger); батч на диалог за такт, ответ в диалог (без reply-to сообщения); вложения по размеру (мал.→файл, кр.→указание, задел веб-диск); управление вкл/выкл письмом (TKT-8).

## TKT-TMB-11 — Двухъящичная модель (уточнение из ревью 12.07; часть-БТ + часть-эксплуатации)
**Контекст.** В игре ДВА ящика, а req смешивал их в «целевой ящик» — неочевидно, в use-cases не разведено.
**TO BE.**
- **Ящик моста (B):** выделенный аккаунт, которым ВЛАДЕЕТ мост. С него мост шлёт (SMTP) доставки TG→почта; его же поллит (IMAP) на ответы и команды пользователя. Требует отдельной настройки (app-пароль) — часть эксплуатации.
- **Ящик пользователя (U):** личный ящик, куда мост ДОСТАВЛЯЕТ (адрес назначения) и откуда пользователь читает и отвечает.
- **Поток:** мост (B) → письмо на U; пользователь читает на U, отвечает с U → ответ приходит на B; мост поллит B → постит в TG.
- **Auth (смыкается с TKT-7/TKT-8):** мост действует по ответу/команде ТОЛЬКО если отправитель = U (доверенный), письмо пришло на B, in-reply-to доставленного.
- **Конфиг:** креды ящика B (IMAP/SMTP) + адрес U (доставка + ожидаемый отправитель ответов). Деплой-README: как завести B.
**Критерий приёмки.** Доставка уходит с B на U; ответ с U на B двигает TG; ответ/команда с чужого адреса игнорируется; оба ящика (B-креды, U-адрес) заданы в конфиге.
**Изменение БД / API / интеграции.** БД: нет. API: IMAP/SMTP на аккаунте B. Интеграции: почтовый провайдер аккаунта B.
**Релиз.** v0.1.

### Уточнение TKT-9 (Евгений 12.07): личка 1:1 — deep-link НЕ даётся (Telegram не поддерживает шаримую ссылку на сообщение в 1:1). Публичные (username) и посты каналов — ссылка есть; приватные группы/каналы участнику — t.me/c/. Личка 1:1: best-effort tg:// или без ссылки; содержание доставляется всегда.

## TKT-TMB-12 — Профиль нагрузки (пропущенное требование, находка живого теста 13.07)
**AS IS.** req/spec не задавали условий нагрузки; lean-прогон свернул load-profile → реализация фетчила ВСЕ диалоги за такт → таймаут на реальном аккаунте (806 личек).
**TO BE.** Явное требование к нагрузке: ожидаемый порядок числа диалогов (сотни-тысячи), темп сообщений, и НЕФУНКЦИОНАЛЬНОЕ: поллинг обязан масштабироваться сублинейно к числу диалогов (фетчить только активные — по top-id; не round-trip на каждый). + нагрузочный/scale-тест в контур (N=1000 диалогов, замер такта).
**Критерий приёмки.** Такт inbound на N=1000 диалогов (из них k активных) укладывается в интервал поллинга; сетевых round-trip ≈ O(k), не O(N).
**Релиз.** v0.1 (фикс top-id уже внесён; требование+тест — доописать).

## TKT-TMB-13 — Политика обработки ботов (Евгений 13.07)
**AS IS.** v0.1 бриджит личку ЛЮБОГО отправителя, включая ботов (bot=True) — бот-нотификации потенциально шумят в почте.
**TO BE.** Конфиг `bot_policy = none | all | selected` (+ `bot_list` для selected): none — сообщения от ботов не бриджим; all — бриджим все; selected — только от ботов из списка. Аналогично mention-политике, но по признаку bot.
**Критерий приёмки.** Сообщение из личка с bot=True обрабатывается по политике: none → игнор, selected → только если бот в bot_list, all → как обычная личка.
**Релиз.** v0.2 (v0.1 = all, текущее поведение).

## TKT-TMB-14 — Копия отправленного в Sent ящика B (Евгений 13.07)
**AS IS.** SMTP-отправка не оставляет следа на сервере B → пользователь не может проверить, что и куда реально ушло (диагностика пропажи письма буксовала).
**TO BE.** Каждое письмо, отправленное мостом (доставки + уведомления), зеркалится в папку Sent ящика B по IMAP APPEND. Настройка `save_sent_copy` (дефолт ВКЛ), имя папки `sent_folder` (дефолт `Sent`). Best-effort: сбой архивации логируется и НИКОГДА не роняет доставку.
**Критерий приёмки.** При включённой настройке отправленное письмо появляется в Sent ящика B; при выключенной — IMAP не открывается; сбой IMAP не срывает отправку.
**Релиз.** v0.1.

## Уточнение TKT-3/TKT-10 (срезка цитаты в ответе, находка живого теста 13.07)
**Дефект.** Ответ из почты постился в TG вместе с процитированным оригиналом («хвостом»): срезка ловила только англ. `On … wrote:` и `>`, а (а) русский клиент даёт другой маркер атрибуции («… писал(а):», дата-префикс) и (б) HTML-ответ схлопывался в одну строку — построчная срезка не срабатывала.
**TO BE.** Устойчивая срезка хвоста: маркеры атрибуции нескольких локалей (`wrote:`, `писал(а):`, баннеры пересылки), правила-разделители; HTML — сохранять переводы строк по блочным тегам и целиком отбрасывать контейнер цитаты (`<blockquote>`, gmail_quote/cite). Постим только чистый top-post пользователя. Bottom-posting в v1 не поддержан.
**Критерий приёмки.** Reply из русского/английского, plain/HTML клиента → в TG уходит только текст пользователя без цитаты.

## Заметка по deliverability (находка живого теста 13.07, не тикет — эксплуатация)
Пропажа письма на eparfenov оказалась НЕ багом моста (SMTP-отправка+композиция проверены — письмо B→arete@noomarxism.ru дошло корректно), а тихим спам-дропом на приёмной стороне (голый subject + внешняя ссылка). Для надёжной доставки продукта: (1) subject с именем отправителя (сделано); (2) **SPF/DKIM/DMARC на домене ящика B** — иначе приёмные серверы (gmail/yandex/корп) тихо режут; в README деплоя — раздел «настрой SPF/DKIM для B, иначе часть писем не дойдёт молча».
